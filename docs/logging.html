日志管道Logging Pipeline

应用日志经 RabbitMQ 采集进 MongoDB，再由 opsbffsrv 检索——一条"日志即消息"的异步管道，应用侧只挂一个 logging.Handler 即可接入。 Application logs are collected over RabbitMQ into MongoDB, then queried back through opsbffsrv — a "logs as messages" async pipeline that an app joins by attaching a single logging.Handler.

整条链路The end-to-end pipeline

采集（生产者）与入库（消费者）解耦在一个 RabbitMQ topic 交换机两侧。Collection (producer) and ingestion (consumer) are decoupled across a RabbitMQ topic exchange.

交换机是 topic 类型。生产者按模板 logs.business.{service}.{level} 生成 routing key（{level} 取小写，如 logs.business.myservice.info）；infrasrv 的消费队列用通配 logs.business.# 绑定，因此能收下所有服务、所有级别的日志。生产与消费两端彻底解耦——应用不阻塞、不直连 Mongo。 The exchange is a topic exchange. The producer builds the routing key from the template logs.business.{service}.{level} ({level} lowercased, e.g. logs.business.myservice.info); infrasrv's consumer queue binds with the wildcard logs.business.#, so it receives logs from every service at every level. Producer and consumer are fully decoupled — the app never blocks and never talks to Mongo directly.

MongoDB 存储MongoDB storage

入库为 LogEventDocument，存到 infra_logging 库的 service_logs 集合。Ingested as LogEventDocument into the service_logs collection of the infra_logging database.

LogEventDocument 在 LogEventCreate 全部字段之上多两个：metadata_flattened（把 metadata 拍平为字符串列表，供全文检索）与 created_at（入库时间）。集合上建有这些索引：(service_name, timestamp↓)、(level, timestamp↓)、(ip, timestamp↓)、(func_name, timestamp↓)，以及在 (message, metadata_flattened) 上的全文索引。 LogEventDocument adds two fields on top of all LogEventCreate fields: metadata_flattened (metadata flattened into a string list for full-text search) and created_at (ingestion time). The collection carries these indexes: (service_name, timestamp↓), (level, timestamp↓), (ip, timestamp↓), (func_name, timestamp↓), plus a text index on (message, metadata_flattened).

过期是调度清理，不是 TTL 索引。当前实现没有给集合挂 expireAfterSeconds 的 TTL 索引，而是由 infrasrv 的定时任务按 LOG_RETENTION_DAYS（默认 7 天）批量删除过期日志，每批上限由 LOG_RETENTION_BATCH_LIMIT（默认 2000）控制。效果同样是"自动过期"，只是落在应用层调度。 Expiry is scheduled cleanup, not a TTL index. The current implementation attaches no expireAfterSeconds TTL index; instead an infrasrv scheduled task batch-deletes expired logs by LOG_RETENTION_DAYS (default 7), each batch capped by LOG_RETENTION_BATCH_LIMIT (default 2000). The net effect is still automatic expiry, just driven at the application layer.

查询 APIQuery API

两类日志走两个接口：业务日志查 MongoDB 管道，服务日志查 PostgreSQL 动作记录。Two log kinds, two endpoints: business logs query the MongoDB pipeline, service logs query PostgreSQL action records.

查询时间窗有上限。业务日志检索受 LOG_QUERY_MAX_RANGE_DAYS（默认 7 天）约束，避免一次拉穿大表。这正是前端 LogSearch 页背后的接口。 Query windows are bounded. Business-log search is constrained by LOG_QUERY_MAX_RANGE_DAYS (default 7 days) to avoid scanning the whole table at once. This is exactly the endpoint behind the frontend LogSearch screen.
