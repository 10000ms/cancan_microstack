工作流引擎 · 节点类型Workflow Engine · Node Types

引擎提供 8 种节点，覆盖"起点 / 调外部 / 变数据 / 走分支 / 并行 / 汇合 / 循环 / 终点"。每种节点对应一个处理器，按类型动态分发；本页逐一讲清它的配置与输出。 The engine ships 8 node types covering "start / call out / reshape / branch / fork / join / loop / end". Each maps to a handler dispatched by type; this page walks through every one's config and output.

下游怎么算出来的？多数节点直接走 next_node_ids；但 LOGIC（按条件二选一）、FORK（一对多）、LOOP（回循环体或走退出）由引擎的 _determine_next_nodes 专门计算。这也是为什么这三种节点的"关键配置"里都带着下游节点 ID。 How is the next node computed? Most nodes simply follow next_node_ids; but LOGIC (pick one of two by condition), FORK (one-to-many) and LOOP (re-enter the body or take the exit) are computed specially by the engine's _determine_next_nodes. That is why these three carry downstream node IDs in their config.

START — 起点— Entry point

每条工作流恰好一个起点，承接触发上下文。Exactly one entry point per workflow, receiving the trigger context.

触发工作流时，应用层找到 START 节点，把它的 next_node_ids 作为首批节点派发出去，运行随即转入 RUNNING。START 处理器本身不做业务，只回显 run_id 与 loop_index 表示"已接收触发上下文"。可选的 trigger_rules 描述触发条件分组（字段 / 操作符 / 值，支持 AND / OR 嵌套）。 On trigger, the application layer locates the START node and dispatches its next_node_ids as the first batch, flipping the run to RUNNING. The START handler does no business work itself — it echoes run_id and loop_index to confirm "trigger context accepted". The optional trigger_rules describe a group of trigger conditions (field / operator / value, with nested AND / OR).

START / END 的节点 ID 是保留字（如 start、node-start、end、node-end），前后端都应视为不可改名。START / END node IDs are reserved words (e.g. start, node-start, end, node-end) — both frontend and backend should treat them as un-renamable.

ACTION — 调外部— Call out

工作流与外界打交道的主力节点：发一个 HTTP 请求，把响应写回上下文。The workhorse for talking to the outside world: fire one HTTP request and write the response back into context.

ACTION 节点的核心是 request 配置。处理器在发出前会用 Jinja2 模板渲染 url / headers / params / 请求体——也就是说这些字段都能引用上下文里的数据。请求体支持多形态：body_type 为 auto 时自动按 json_body / form_body / raw_body 与 Content-Type 推断 JSON / 表单 / multipart / 原始体。 The heart of an ACTION node is its request config. Before firing, the handler renders url / headers / params / body with Jinja2 — meaning all of them can reference context data. The body is polymorphic: with body_type = auto the engine infers JSON / form / multipart / raw from json_body / form_body / raw_body and the Content-Type.

配置Config说明Detail
request.url / method目标地址与方法（默认 GET），url 经 Jinja2 渲染。Target URL and method (default GET); url is Jinja2-rendered.
request.headers / params请求头与查询参数，逐个经模板渲染。Headers and query params, each template-rendered.
request.body / json_body / form_body / raw_body请求体多形态来源；配合 body_type（auto / json / form-urlencoded / multipart / raw）选定发送方式。Polymorphic body sources; combined with body_type (auto / json / form-urlencoded / multipart / raw) to decide how it is sent.
request.timeout_seconds请求超时，默认 30 秒。Request timeout, default 30s.
async_mode为真时不等结果：注入 X-Callback-Url，节点转 SUSPENDED 等外部回调。When true, do not wait: inject X-Callback-Url and move the node to SUSPENDED awaiting an external callback.
context_mappings把响应字段按点路径映射到 global_context 业务键，如 {"biz.user_id": "json_body.data.user_id"}。Maps response fields by dot path into business keys in global_context, e.g. {"biz.user_id": "json_body.data.user_id"}.
retry_policy · failure_strategy可选重试（次数 / 间隔 / 按状态码）与失败处理策略。Optional retry (attempts / interval / on-status) and failure-handling strategy.

同步模式下节点返回 SUCCESS；异步模式（async_mode=true）返回 SUSPENDED，直到回调到达才继续。In sync mode the node returns SUCCESS; in async mode (async_mode=true) it returns SUSPENDED until the callback arrives.

// ACTION 节点配置示例 / ACTION node config example
{
  "async_mode": false,
  "request": {
    "method": "POST",
    "url": "https://api.internal/users/{{ context.user_id }}/notify",
    "headers": { "Authorization": "Bearer {{ context.token }}" },
    "json_body": { "channel": "email", "order_id": "{{ context.order_id }}" },
    "timeout_seconds": 30
  },
  "context_mappings": { "biz.notify_id": "json_body.data.id" }
}

TRANSFORM — 变数据— Reshape data

在节点之间整形数据：把上游零散字段提取、改名、拼成下游需要的形状。Reshape data between nodes: extract, rename and assemble scattered upstream fields into the shape the next node needs.

TRANSFORM 不调外部，只在上下文里做数据搬运。它按 output_schema 逐键求值：每个目标键对应一个表达式，由 engine 决定用 JMESPath（适合从 JSON 里按路径 / 过滤提取）还是 Jinja2（适合拼字符串、做简单运算），求值结果汇成一个新 dict 作为输出。 TRANSFORM calls nothing external; it only moves data within the context. It evaluates the output_schema key by key: each target key maps to an expression, and engine decides whether to use JMESPath (great for path / filter extraction from JSON) or Jinja2 (great for string assembly and simple arithmetic). The results form a new dict as the output. 

// TRANSFORM 节点配置示例（JMESPath 引擎）/ TRANSFORM example (JMESPath engine)
{
  "engine": "JMESPATH",
  "output_schema": {
    "vip_user_ids": "__runtime__.last_output.json_body.data[?vip==`true`].id",
    "total":        "__runtime__.last_output.json_body.data | length(@)"
  }
}

LOGIC — 条件分支— Conditional branch

二选一：根据一个条件表达式决定走 true 路还是 false 路。One of two: a single condition expression decides whether to take the true path or the false path.

LOGIC 的 condition 是 Jinja2 表达式（不带 {{ }} 时引擎会自动补上）。渲染结果按布尔规则求值后：为真走 true_next_node_id，为假走 false_next_node_id。处理器输出里会回显 result 与 selected_branch（TRUE / FALSE），方便在运行图上看清"当时走了哪条"。 LOGIC's condition is a Jinja2 expression (the engine auto-wraps it in {{ }} if omitted). After the rendered result is evaluated as a boolean: true takes true_next_node_id, false takes false_next_node_id. The handler echoes result and selected_branch (TRUE / FALSE) so the run graph shows "which way it went". 

// LOGIC 节点配置示例 / LOGIC node config example
{
  "condition": "context.__runtime__.last_output.status_code == 200",
  "true_next_node_id":  "node_send_receipt",
  "false_next_node_id": "node_alert_ops"
}

FORK — 并行分发— Parallel fan-out

把流程一分为多：同时派发多个分支，各自独立执行。Split the flow into many: dispatch several branches at once, each running independently.

FORK 本身不做业务，只是一个语义标记。它把 branch_node_ids 列表里的所有节点作为下游一次性派发；调度层会为每个分支创建独立的执行任务并行推进。处理器输出回显 branch_node_ids 与 branch_count。分支跑完后通常用 JOIN 汇合。 FORK does no business work; it is a semantic marker. It dispatches every node in branch_node_ids as downstream at once, and the scheduling layer creates an independent execution task per branch to run them in parallel. The handler echoes branch_node_ids and branch_count. Branches usually reconverge at a JOIN.

JOIN — 分支汇合— Branch convergence

把多个并行分支收束回一条线，按 ALL / ANY 决定何时放行。Collapse parallel branches back into one line, releasing on ALL or ANY semantics.

JOIN 处理器需要工作流定义来反查"谁指向我"（上游节点）。它检查这些上游节点在当前上下文里的状态： The JOIN handler needs the workflow definition to look up "who points at me" (upstream nodes). It inspects those upstream nodes' statuses in the current context:

  • mode = ALL：所有上游都为 SUCCESS 或 SKIPPED 才就绪放行。mode = ALL: ready only when every upstream is SUCCESS or SKIPPED.
  • mode = ANY：只要有一个上游 SUCCESS 即就绪。mode = ANY: ready as soon as any one upstream is SUCCESS.

未就绪时节点返回 PENDING（不向下游派发，等其余分支到齐再被重新评估）；就绪则返回 SUCCESS 并继续。输出含 upstream_statuses 与 is_ready。 When not ready it returns PENDING (no downstream dispatch; it is re-evaluated as remaining branches arrive); when ready it returns SUCCESS and continues. The output includes upstream_statuses and is_ready.

LOOP — 循环控制— Loop control

重复执行一段循环体，直到退出条件成立或达到最大轮数。Repeat a loop body until the exit condition holds or the iteration cap is hit.

LOOP 是一个控制节点。每轮它求值退出条件 condition（Jinja2）：未退出则把下游指向循环体入口 body_entry_id；退出则走 exit_node_id（或节点自身的 next_node_ids）。当前轮次由上下文的 loop_index 跟踪。 LOOP is a control node. Each round it evaluates the exit condition (Jinja2): if not exiting, it points downstream at the body entry body_entry_id; if exiting, it takes exit_node_id (or the node's own next_node_ids). The current round is tracked by the context's loop_index.

配置Config说明Detail
condition退出条件表达式；为真即退出循环。Exit condition expression; true means leave the loop.
body_entry_id循环体入口节点 ID（每轮回到这里）。The loop body entry node ID (returned to each round).
exit_node_id退出后去向的节点 ID。The node ID to go to after exit.
max_iterations最大轮数（默认 10），到顶强制退出，防止死循环。Iteration cap (default 10); a forced exit at the top prevents infinite loops.
jump_target_id兼容字段，等价于 body_entry_id（历史数据会自动同步）。Backward-compatible alias of body_entry_id (legacy data is auto-synced).
边界轮次防误差。当退出条件恰好在 loop_index == max_iterations 才变真时，引擎会做一次"上一轮探测"，允许本轮循环体再执行一次，避免出现"max=3 却只跑了 2 次"的差一错误。此外，循环体末端节点可以省略显式回边——引擎会自动把它回派给 LOOP 控制节点并 loop_index + 1。 Boundary-round guard. When the exit condition flips true only at loop_index == max_iterations, the engine probes the previous round and allows the body to run once more, avoiding the "max=3 but only 2 runs" off-by-one. Also, a loop-body tail node may omit its explicit back-edge — the engine auto-dispatches it back to the LOOP control node with loop_index + 1.

END — 终点— Terminal

收尾节点：决定整条运行最终是成功还是失败。The closing node: it decides whether the whole run ends as success or failure.

END 的 status 取 SUCCESS 或 FAILURE。当 END 以 SUCCESS 收尾，应用层把整条运行标记为 SUCCESS；反之标记 FAILURE。config 缺省时按 SUCCESS 处理。可选的 output_schema 用于约定收尾输出形状。 END's status is either SUCCESS or FAILURE. When END finishes with SUCCESS, the application layer marks the whole run SUCCESS; otherwise FAILURE. A missing config defaults to SUCCESS. The optional output_schema shapes the final output. 

// END 节点配置示例 / END node config example
{ "status": "SUCCESS" }
