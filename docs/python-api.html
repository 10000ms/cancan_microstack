CancanMicrostack 门面方法CancanMicrostack facade methods

三个高层方法覆盖最常见的三件事：合成 compose、导出资产、启动服务。Three high-level methods cover the three most common jobs: synthesize compose, export an asset, run a service.

从顶层包导入门面即可。构造时它会一次性组装好 AssetManager / ComposeBuilder / ServiceRunner，你只需认识一个对象。 Import the facade from the top-level package. Constructing it wires up AssetManager / ComposeBuilder / ServiceRunner at once, so you only deal with a single object. 

from cancan_microstack import CancanMicrostack

stack = CancanMicrostack()

build_compose(*, workspace, service_file=None, overrides=None, output=None) -> Path

以内置 infra compose 为 base，依次深度合并业务 compose 与多份 overrides，写出最终栈文件并返回其路径。output 为空时默认落到 workspace/compose.cancan.yml。注意全部是关键字参数。 Starting from the built-in infra compose, it deep-merges the business compose and any overrides in order, writes the final stack file and returns its path. When output is omitted it defaults to workspace/compose.cancan.yml. Note that all arguments are keyword-only. 

from pathlib import Path

out = stack.build_compose(
    workspace=Path("."),
    service_file=Path("docker-compose.services.yml"),
    overrides=[Path("compose.dev.yml")],
)
print(out)  # .../compose.cancan.yml

export_asset(logical_name, destination, overwrite=False) -> Path

把一个包内资产复制到工作区路径并返回目的地。目录用 copytree、文件用 copy2；目的地已存在且 overwrite=False 时抛 FileExistsError。这是让 Docker 能挂载到资产的标准做法。 Copies a packaged asset to a workspace path and returns the destination. Directories use copytree, files use copy2; if the destination exists and overwrite=False it raises FileExistsError. This is the canonical way to make assets mountable by Docker. 

stack.export_asset(
    "docker/docker-compose.infra.yml",
    Path("infra/docker-compose.infra.yml"),
    overwrite=True,
)

run_service(service_name, host="0.0.0.0", port=8080, workspace=None) -> None

按服务名（controllersrv / infrasrv / opsbffsrv）启动内置服务：内部会先 bootstrap 工作区与覆盖目录，再动态 import 入口点并调用其 main(host, port)。这是阻塞调用（内部 asyncio.run）；要在已有事件循环里跑，用 await stack.run_service_async(...)。 Starts a bundled service by name (controllersrv / infrasrv / opsbffsrv): it bootstraps the workspace and override dir, then dynamically imports the entrypoint and calls its main(host, port). This call blocks (it wraps asyncio.run); inside an existing event loop use await stack.run_service_async(...). 

stack.run_service("controllersrv", port=22100)  # blocks until the service stops
端口约定：方法签名默认 8080，但 CLI 对 controllersrv 默认用 22100。代码里启动 controllersrv 时建议显式传 port=22100，与整栈保持一致。 Port convention: the signature defaults to 8080, but the CLI defaults controllersrv to 22100. When starting controllersrv from code, pass port=22100 explicitly to stay consistent with the full stack.

完整使用例Full example

合成栈 → 导出资产 → 启动服务，一气呵成。Synthesize the stack, export an asset, then run a service — end to end.

from pathlib import Path
from cancan_microstack import CancanMicrostack

stack = CancanMicrostack()
workspace = Path(".").resolve()

# 1) 合成最终 compose 栈 / synthesize the final compose stack
compose_file = stack.build_compose(
    workspace=workspace,
    service_file=workspace / "docker-compose.services.yml",
    overrides=[workspace / "compose.dev.yml"],
)

# 2) 导出包内资产，供 Docker 挂载 / export a packaged asset for Docker to mount
stack.export_asset(
    "ddl/create_db.sql",
    workspace / "ddl" / "create_db.sql",
    overwrite=False,
)

# 3) 在宿主机启动 controllersrv（阻塞）/ run controllersrv on the host (blocking)
stack.run_service("controllersrv", port=22100, workspace=workspace)

AssetManager 直接用法Using AssetManager directly

只想读/导出资产，不必经过门面，可以单独实例化 AssetManager。If you only need to read or export assets, instantiate AssetManager on its own.

from pathlib import Path
from cancan_microstack import AssetManager

assets = AssetManager()

# 列出某子目录下的资产 / list assets under a subdir
for rec in assets.list_assets("docker"):
    print(rec.logical_name, rec.is_dir)

# 导出单个资产 / export one asset
assets.export_asset("docker/docker-compose.infra.yml",
                    Path("infra/docker-compose.infra.yml"))

# 直接读文本 / read text directly
yaml_text = assets.read_text("docker/docker-compose.infra.yml")

# 物化成可访问的 Path / materialize a usable Path
path = assets.resolve_path("ddl/create_db.sql")
  • list_assets(subdir=None) — 递归返回 AssetRecord(logical_name, path, is_dir) 列表；不传 subdir 则列全部。recursively returns a list of AssetRecord(logical_name, path, is_dir); lists everything when subdir is omitted.
  • export_asset(logical_name, destination, overwrite=False) — 同门面方法，复制资产到工作区。same as the facade method, copies the asset into the workspace.
  • read_text(logical_name) — 以 UTF-8 读出文本内容。reads text content as UTF-8.
  • resolve_path(logical_name) — 物化资产并返回 Path，资产不存在时抛 FileNotFoundError。materializes the asset and returns a Path, raising FileNotFoundError if it does not exist.

导入风格Import style

这一条会影响你怎么写 import，值得单独记住。This one rule shapes how you write imports — worth remembering on its own.

从真实定义模块直接 import，不依赖 __init__ re-export。本库的 public/*/__init__.py 多数是空的命名空间，并不集中转发符号。请按上例那样从 cancan_microstack.public.schemas.common 这类具体模块导入，而不是指望 from cancan_microstack.public import APIResponse。唯一例外是顶层包：CancanMicrostack / AssetManager / ComposeBuilder / ServiceRunner 确实在 cancan_microstack/__init__.py 做了导出，可直接 from cancan_microstack import ...。 Import straight from the defining module; do not rely on __init__ re-exports. Most public/*/__init__.py files here are empty namespaces and do not forward symbols. Import from the concrete module like cancan_microstack.public.schemas.common as shown, rather than expecting from cancan_microstack.public import APIResponse. The one exception is the top-level package: CancanMicrostack / AssetManager / ComposeBuilder / ServiceRunner are re-exported in cancan_microstack/__init__.py, so from cancan_microstack import ... works.
