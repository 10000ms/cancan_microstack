健康检查机制Health-check mechanism

被动心跳兜底"还活着"，主动探测确认"还健康"，连续失败触发自动重启，操作窗口内予以豁免。Passive heartbeat catches "still alive", active probing confirms "still healthy", repeated failures trigger auto-restart, and an operation window grants exemption.

机制Mechanism依据字段 / 常量Field / constant行为Behavior
被动心跳Passive heartbeat last_heartbeat
INSTANCE_HEARTBEAT_TIMEOUT_SECONDS=90		 实例自报心跳；超 90 秒未更新即判为僵尸候选。Instances self-report; no update for 90s flags it as a zombie candidate.
主动探测Active probe health_check_path（/internal/health）
HEALTH_CHECK_TIMEOUT=10s · RETRY=2		 HTTP 探测实例 host:port 的健康路径，期望 200，失败重试 2 次（间隔 1s）。HTTP-probes the instance's host:port health path, expecting 200, retrying twice (1s apart) on failure.
自动重启Auto-restart consecutive_failures
CONSECUTIVE_FAILURE_THRESHOLD=3		 连续失败达 3 次 → 标 UNHEALTHY 并触发重启（回调 controllersrv 的异步重启接口）。Three consecutive failures → mark UNHEALTHY and trigger a restart (callback to controllersrv's async-restart endpoint).
窗口豁免Window exemption OPERATION_WINDOW_MINUTES=5 若该实例 5 分钟内有进行中 / 刚完成的运维操作（如重启），探测予以豁免，health_status=exempted，避免误判。If the instance had an in-progress / just-completed op (e.g. restart) within 5 minutes, probing is exempted with health_status=exempted to avoid false alarms.

注册钩子链Registration hook chain

注册 / 注销前后插入可扩展校验与加工，按优先级串行执行，结果决定链路是否继续。Extensible validation and enrichment around register / deregister, run serially by priority, with the result deciding whether the chain proceeds.

钩子类型分 PRE_REGISTER / POST_REGISTER / PRE_DEREGISTER（及对应的 POST_DEREGISTER）。同一阶段的钩子按优先级值升序串行执行（值越小越先；内置档位 HIGHEST=0 / HIGH=25 / NORMAL=50 / LOW=75 / LOWEST=100），全局钩子先于按服务类型的钩子。 Hook types are PRE_REGISTER / POST_REGISTER / PRE_DEREGISTER (plus the matching POST_DEREGISTER). Hooks in a stage run serially in ascending priority order (lower runs first; built-in tiers HIGHEST=0 / HIGH=25 / NORMAL=50 / LOW=75 / LOWEST=100), with global hooks ahead of service-type-specific ones.

结果语义Result semantics

结果Result链路动作Chain action
SUCCESS继续执行下一个钩子。Continue to the next hook.
FAILURE记警告并继续（可在 max_retries 内重试，默认 2）。Log a warning and continue (retrying within max_retries, default 2).
TERMINATE立即终止，拒绝注册并返回 HTTP 400（错误码 REGISTRATION_REJECTED）。Stop immediately, reject registration and return HTTP 400 (error code REGISTRATION_REJECTED).
SKIP / RETRY跳过当前钩子继续 / 请求重试当前钩子。Skip this hook and continue / request a retry of this hook.
内置钩子示例。服务名格式校验、Host 校验（拒绝 localhost / 127.0.0.1 / 0.0.0.0）、元数据加工（补 registered_at、推断 service_type）、端口范围校验（1024–65535）、单服务实例配额。这些都是普通钩子，可按需扩展。 Built-in hooks (examples). Service-name format check, host check (reject localhost / 127.0.0.1 / 0.0.0.0), metadata enrichment (add registered_at, infer service_type), port-range check (1024–65535), and per-service instance quota. These are ordinary hooks you can extend.

僵尸实例清理Zombie-instance cleanup

心跳超时先软删，下沉一段时间后再硬删；连续探测失败则标 DOWN 触发重启。Heartbeat timeout soft-deletes first, then hard-deletes after settling; repeated probe failures mark DOWN and trigger a restart.

  1. 阶段一：软删僵尸Phase 1: soft-delete zombies

    仍为 UP 但 last_heartbeat 超 90 秒（或创建超时）的实例，软删——置 flag=1 并 status=DOWN，记原因"Stale heartbeat"。Instances still UP but with last_heartbeat over 90s stale (or creation-timed-out) are soft-deleted — set flag=1 and status=DOWN, reason "Stale heartbeat".

  2. 阶段二：硬删下沉实例Phase 2: hard-delete settled instances

    已 DOWN 且 update_time 再超 90 秒的实例，物理删除，彻底出表。Instances already DOWN whose update_time is a further 90s stale are physically deleted, fully leaving the table.

  3. 探测失败路线Probe-failure path

    与心跳并行：主动探测连续失败超阈值（3 次）的实例标 UNHEALTHY 并触发自动重启——区别于心跳超时的"无信号"路线。In parallel with heartbeats: an instance whose active probes fail past the threshold (3) is marked UNHEALTHY and auto-restarted — distinct from the "no signal" heartbeat path.

服务配置同步Service-config sync

配置写在中心表里，再由 push 广播给该服务的活跃实例热加载。Config is written to the central table, then push-broadcast to the service's live instances for hot reload.

方法Method路径Path用途Purpose
GETservice_config按 service_name 取配置Fetch config by service_name
GETservice_configs取全部服务的配置Fetch every service's config
POST / PUT / DELETEservice_config新增 / 更新 / 删除配置键Create / update / delete config keys
POSTinternal/config/push把某服务配置广播下发给其活跃实例Broadcast a service's config to its live instances

配置更新后调用 internal/config/push：infrasrv 取出该服务在 service_config_tbl 的全部键值，拼成配置字典，逐个 POST 到各活跃实例的 /internal/config/update 让其热加载。返回分实例的部分成功结果（总数 / 成功 / 失败 + 每实例明细）；个别下发失败仅记警告，不会立即把实例标 DOWN，留待下一轮健康检查处理。 After a config update, call internal/config/push: infrasrv reads all of that service's key-values from service_config_tbl, assembles a config dict, and POSTs it to each live instance's /internal/config/update for hot reload. It returns a per-instance partial-success result (total / success / failed + per-instance detail); individual push failures are logged as warnings only — they do not immediately mark an instance DOWN, leaving that to the next health-check cycle.
