通用约定Shared conventions

先理解所有表共有的标准字段与初始化方式，下面的表格只列各自的关键业务字段。Understand the standard fields and initialization model shared by every table; the tables below list only each one's key business columns.

标准字段。每张表都带三个标准字段：flag（SMALLINT，0=正常、1=软删，业务删除走 flag 而非物理删除）、created_time（创建时间）与 update_time（最后更新时间）。每张表都挂了一个 BEFORE UPDATE 触发器，行更新时自动把 update_time 刷成 CURRENT_TIMESTAMP，无需应用层维护。表的 DDL 随服务启动由 DDLManager 自动初始化——脚本目录来自 DDL_SCRIPT_PATH（默认指向库内置的 ddl/infra 与 ddl/ops 资产，可被工作区覆盖）。 Standard fields. Every table carries three standard columns: flag (SMALLINT, 0 = active, 1 = soft-deleted — deletes set flag instead of removing the row), created_time, and update_time. Each table also has a BEFORE UPDATE trigger that automatically refreshes update_time to CURRENT_TIMESTAMP on every row update, so the application never maintains it by hand. Table DDL is initialized automatically by DDLManager on service startup — the script directory comes from DDL_SCRIPT_PATH (which defaults to the bundled ddl/infra and ddl/ops assets and can be overridden from the workspace).

下面表格的关键字段列只挑选业务上最有信息量的列，省略了主键 id 与上述三个标准字段。多数业务列用 JSONB 存放可扩展的结构化数据（如元数据、配置、上下文、快照）。 The key fields column below highlights the most informative business columns only; it omits the id primary key and the three standard fields above. Many business columns use JSONB to hold extensible structured data (metadata, config, context, snapshots).

infra 库 — 服务注册与工作流引擎infra database — registry & workflow engine

11 张表：前 5 张围绕服务/实例/配置/操作/行为审计，后 6 张构成工作流引擎的定义、版本、运行、节点、执行日志与告警。Eleven tables: the first five cover services / instances / config / operations / action audit; the last six form the workflow engine — definition, version, run, node, execution log and alerts.

表名Table 用途Purpose 关键字段Key fields
service_info_tbl 服务元信息与期望规模：每个服务一条记录，记录健康检查路径、期望/实际副本数与扩缩容策略。Service metadata and desired scale: one row per service, with health-check path, desired/actual replica counts and scale policy. service_name, service_type, health_check_path, expected_status, desired_replicas, actual_replicas, scale_policy, service_metadata
service_instance_tbl 服务实例运行态：每个容器/进程实例一条，记录地址端口、运行状态、健康状态、心跳与失败计数。Per-instance runtime state: one row per container/process instance with address, status, health, heartbeat and failure counts. instance_id, service_name, host, port, status, health_status, last_heartbeat, consecutive_failures, restart_count
service_config_tbl 服务级键值配置：按 (service_name, conf_key) 唯一存放可下发的运行时配置项。Service-level key/value config: pushable runtime settings, unique on (service_name, conf_key). service_name, conf_key, conf_value
service_operation_tbl 异步操作记录：start/stop/restart 等操作的状态机、入参、结果与重试，按 operation_id 唯一。Async operation records: the state machine, params, result and retries of start/stop/restart operations, unique on operation_id. operation_id, operation_type, service_name, status, operation_params, result, retry_count, max_retries, initiated_by
service_action_log_tbl 服务行为审计日志：记录注册、注销、心跳、健康失败、重启、扩缩容、重建等关键动作。Service action audit log: register, deregister, heartbeat, health-check failures, restart, scale and rebuild events. service_name, instance_id, action_type, action_status, action_detail, error_message, triggered_by
workflow_definition_tbl 工作流定义：工作流的静态结构与配置，含画布数据、扁平节点 Map、Cron 计划、启用开关与版本号。主键为 UUID。Workflow definition: the static structure and config — graph data, a flattened node map, a Cron schedule, an active toggle and a version number. UUID primary key. name, schedule, graph_data, nodes_config, global_context, is_active, version, change_summary
workflow_definition_version_tbl 工作流版本快照：每次发布或回滚后保留一份定义快照，按 (workflow_id, version) 唯一，引用定义表。Workflow version snapshot: a definition snapshot kept after each publish or rollback, unique on (workflow_id, version), referencing the definition table. workflow_id, version, name, schedule, graph_data, nodes_config, change_summary
workflow_run_tbl 工作流运行实例：一次执行一条，记录触发来源、运行状态、全局上下文，并固化触发时的定义版本与快照以保证运行不可变。Workflow run instance: one row per execution, capturing trigger source, run status, global context, plus the definition version and snapshot taken at trigger time to keep runs immutable. workflow_id, status, trigger_type, trigger_context, global_context, definition_version, definition_snapshot, started_at, finished_at, duration_ms
node_instance_tbl 节点运行实例：记录一次运行中每个节点（含循环轮次）的状态、入参与产出，按 (run_id, node_id, loop_index) 唯一。Node run instance: per-node state (with loop iteration), input and output within a run, unique on (run_id, node_id, loop_index). run_id, node_id, loop_index, status, input_data, final_output, attempt_count, error_msg
execution_log_tbl 执行审计日志：节点实例的每一次尝试一条，保存请求/响应快照、结果状态与耗时。Execution audit log: one row per attempt of a node instance, with request/response snapshots, result status and duration. node_instance_id, attempt_no, request_snapshot, response_snapshot, status, error_detail, start_time, end_time, duration_ms
workflow_engine_alert_tbl 工作流引擎告警：运行中产生的告警，含严重级别、分类、确认与解决的状态机。Workflow engine alerts: alerts raised during runs, with severity, category and an acknowledge/resolve state machine. run_id, node_id, loop_index, severity, category, reason, detail, status, acknowledged_by, resolved_by
工作流五表自上而下构成一条引用链：workflow_definition_tbl → workflow_run_tbl → node_instance_tbl → execution_log_tbl；告警与版本快照分别旁挂在 run 与 definition 上。节点类型见 nodes_config：START / ACTION / TRANSFORM / LOGIC / FORK / JOIN / LOOP / END。 The five workflow tables form a top-down reference chain: workflow_definition_tbl → workflow_run_tbl → node_instance_tbl → execution_log_tbl; alerts and version snapshots hang off the run and the definition respectively. Node types live in nodes_config: START / ACTION / TRANSFORM / LOGIC / FORK / JOIN / LOOP / END.

ops 库 — 运维控制面与 Caddy 网关ops database — control plane & Caddy gateway

6 张表：1 张运维后台用户表，5 张 Caddy 网关管理表（路由、限流、证书、统计、访问日志）。Six tables: one ops admin user table, plus five Caddy gateway tables — routes, rate limits, certificates, stats and access logs.

表名Table 用途Purpose 关键字段Key fields
admin_user_tbl 运维后台用户：登录凭证与 TOTP 双因子绑定状态，username 唯一。Ops admin users: login credentials and TOTP two-factor binding state, unique on username. username, password_hash, totp_secret_encrypted, totp_bound
caddy_route_tbl 网关路由：域名 + 路径 → 上游服务的映射，含 HTTPS、WAF、负载均衡、健康检查与优先级配置，route_name 唯一。Gateway routes: domain + path → upstream mapping, with HTTPS, WAF, load-balancing, health-check and priority settings, unique on route_name. route_name, domain, path_pattern, upstream_service, upstream_host, upstream_port, enable_https, enable_waf, load_balance_strategy, priority, is_enabled
caddy_rate_limit_tbl 限流规则：按匹配条件限制请求速率，含限流窗口、突发额度、阻断响应与 IP 黑白名单，rule_name 唯一。Rate-limit rules: throttle requests by match criteria, with window, burst size, block response and IP allow/deny lists, unique on rule_name. rule_name, match_type, match_pattern, limit_type, limit_value, limit_window, limit_key, burst_size, block_status_code, whitelist_ips, blacklist_ips, is_enabled
caddy_certificate_tbl TLS 证书：域名证书与私钥，含有效期、ACME 自动签发/续订配置与续订状态，domain 唯一。TLS certificates: per-domain cert and key, with validity window, ACME auto-issue/renew config and renewal state, unique on domain. domain, alt_domains, certificate_pem, private_key_pem, issuer, issued_at, expires_at, auto_renew, renew_before_days, status, acme_provider, acme_challenge_type
caddy_stats_tbl 网关统计聚合：按时间窗口与维度聚合请求量、流量、响应时延分位、WAF/限流/TLS 计数，唯一索引防重复聚合。Gateway stats rollups: requests, bytes, response-time percentiles and WAF/rate-limit/TLS counts aggregated by time window and dimension; a unique index prevents duplicate rollups. stat_time, stat_period, dimension_type, dimension_value, total_requests, success_requests, server_error_requests, avg_response_time, p95_response_time, p99_response_time, waf_blocked_requests, rate_limited_requests, unique_ips
caddy_access_log_tbl 网关访问日志：逐请求明细，含客户端与 GeoIP 地理信息、请求/响应、匹配路由与上游、WAF/限流/TLS 字段，request_id 唯一。Gateway access log: per-request detail with client and GeoIP geo data, request/response, matched route and upstream, plus WAF/rate-limit/TLS fields, unique on request_id. request_id, timestamp, client_ip, country, method, host, path, matched_route, upstream_service, status_code, response_time, waf_action, rate_limited, tls_version
