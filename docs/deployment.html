依赖与环境Dependencies & environment

一行 pip 装好工具箱，一条命令拉起整栈；中间件由内置 infra compose 提供，无需手装。One pip command for the toolkit, one command for the whole stack; the middleware comes from the bundled infra compose — nothing to install by hand.

要求Requirement说明Detail
Python ≥ 3.10运行库与 CLI 的宿主机环境。Host runtime for the library and CLI.
Docker 或 or Podman运行容器栈所需；CLI 自动探测，可用 --engine 显式指定。Required to run the container stack; auto-detected, overridable via --engine.
PostgreSQL · Redis · MongoDB · RabbitMQ由内置 infra compose 作为容器一并提供，无需单独部署。Provided as containers by the bundled infra compose — no separate deployment.
 
# 安装工具箱 / Install the toolkit
pip install cancan-microstack

# 在业务仓根目录一键起栈 / Bring up the whole stack from your repo root
cancan stack up --workspace . --service-file docker-compose.services.yml

pip install 会一并带上姊妹库 linglong-web[all]（Web 框架，编排 PostgreSQL / Redis / MongoDB / RabbitMQ / Celery）与 dragonfly-container[all]（容器操作封装）。安装即提供 cancan 命令，等价于 python -m cancan_microstack.cli。完整起栈流程见 快速开始。 pip install also pulls in the sibling libraries linglong-web[all] (the web framework that orchestrates PostgreSQL / Redis / MongoDB / RabbitMQ / Celery) and dragonfly-container[all] (the container-control wrapper). Installation provides the cancan command, equivalent to python -m cancan_microstack.cli. See Quick Start for the full bring-up flow.

controllersrv 要操作 Docker/Podman，因此跑在宿主机、不进容器；其余服务都是容器，由内置 infra compose 与你的业务 compose 合成后统一拉起。容器化服务不直接暴露端口，外部一律经 Caddy 网关进入。infra compose 还附带三个数据库可视化工具（pgweb / redis-commander / mongo-express）和 RabbitMQ 管理控制台，均经 opsbffsrv 代理在运维前端中访问。 controllersrv must drive Docker/Podman, so it runs on the host rather than in a container; everything else is a container, brought up together from the bundled infra compose merged with your business compose. Containerized services do not expose ports directly — all external traffic enters through the Caddy gateway. The infra compose also bundles three database visualization tools (pgweb / redis-commander / mongo-express) and the RabbitMQ management console, all reachable in the admin UI via opsbffsrv proxies.

生产安全Production safety

开箱默认值是为开发便利，绝不是为生产准备的——上线前必须用环境变量替换。The out-of-the-box defaults are for development convenience, never for production — replace them with environment variables before going live.

切勿沿用默认凭证。内置 compose 与服务配置使用本地开发的弱默认值便于开箱即用——例如 PostgreSQL 密码 postgres123，RabbitMQ / MongoDB / mongo-express 等的 admin123。这些值全部可被环境变量覆盖。生产部署时务必通过环境变量注入真实凭证，切勿把任何默认值带上线。（注意：opsbffsrv 运维后台的 admin 用户密码在首次启动时随机生成并写入启动日志，需在首次登录后立即修改，而非固定弱口令。） Never ship the default credentials. The bundled compose and service configs use weak local-dev defaults so the stack works out of the box — for example the PostgreSQL password postgres123, and admin123 for RabbitMQ / MongoDB / mongo-express. Every one of them is overridable via environment variables. In production, always inject real secrets through env vars and never carry any default to a live deployment. (Note: the opsbffsrv admin console's admin user gets a randomly generated password on first startup, written to the startup log — change it right after the first login rather than relying on a fixed weak password.)

覆盖与定制Overrides & customization

不必 Fork 整个库——只替换你想改的那几个文件。No need to fork the whole library — replace only the files you actually want to change.

在工作区放一个 cancan_overrides/<service>/ 目录，只放你要替换的文件，运行时通过 resolve_workspace_or_asset 工作区优先加载它，其余仍复用库内置实现。覆盖根可由 CANCAN_OVERRIDE_ROOT 指定。这样升级库时，未覆盖的部分自动随版本演进，你只维护真正定制的那一小部分。 Place a cancan_overrides/<service>/ directory in your workspace holding only the files you want to replace; at runtime resolve_workspace_or_asset loads it workspace-first, falling back to the bundled implementation for everything else. The override root can be set via CANCAN_OVERRIDE_ROOT. This way, when you upgrade the library, everything you did not override evolves with the version automatically, and you maintain only the small slice you genuinely customized.

升级Upgrades

基础设施随库版本演进，而不是到处 copy-paste 模板。Infrastructure evolves with the library version, not by copy-pasting templates around.

# bump 版本并重新安装 / Bump the pinned version and reinstall
pip install --upgrade cancan-microstack

compose、DDL、脚本与运维前端都是包内资产，随库一起发布。升级库即升级基础设施：bump 版本 → 重新 pip install → 下次 cancan stack up 用上新的内置资产。你只需关心被覆盖的文件是否需要随新版本调整。 compose, DDL, scripts and the ops UI are all in-package assets shipped with the library. Upgrading the library upgrades the infrastructure: bump the version → re-run pip install → the next cancan stack up uses the new bundled assets. You only need to check whether any files you overrode need adjusting for the new version.
