工作流引擎 · 概念与状态机Workflow Engine · Concepts & State Machine

在写第一个工作流之前，先理解引擎赖以运转的五个对象——定义、版本、运行、节点、日志——以及它们如何在状态机里流转。这一层模型是后面节点、调度、API 三章的共同地基。 Before authoring your first workflow, understand the five objects the engine is built on — definition, version, run, node, log — and how they move through their state machines. This model is the shared foundation for the nodes, scheduling and API chapters that follow.

分层概念模型The layered concept model

从"画出来的图"到"跑起来的实例"，引擎用五层对象把设计与运行彻底分开。From "the graph you draw" to "the instance that runs", the engine separates design from runtime with five layers of objects.

设计态只有两层：你编辑的工作流定义，以及每次保存时自动留存的版本快照。运行态有三层：一次运行实例、运行中产生的若干节点实例、以及每个节点每次尝试的执行日志。它们各自落在独立的存储表里，互不覆盖。 The design side has two layers: the workflow definition you edit, and the version snapshot automatically kept on every save. The runtime side has three: a single run, the several node instances it spawns, and the execution log of every attempt of every node. Each lives in its own storage table and never overwrites another.

一句话记住层级。一个 定义 有多个 版本；一次触发产生一个 运行；一个运行展开成多个 节点实例；每个节点实例的每次尝试写一条 执行日志。越往下越细，越往下越多。 The hierarchy in one line. One definition has many versions; one trigger creates one run; one run unfolds into many node instances; each attempt of each instance writes one execution log. The deeper you go, the finer and more numerous.

执行上下文：节点之间传数据Execution context: passing data between nodes

WorkflowExecutionContext 是节点处理器拿到的唯一入参，承载整条运行的状态。WorkflowExecutionContext is the single argument every node handler receives — it carries the state of the whole run.

每个节点执行时，应用层会从运行实例重建一份执行上下文交给领域引擎。节点读它取输入、写 global_context 留输出，下游节点再读到。其中 global_context.__runtime__ 是引擎保留的运行时命名空间，节点自己不要往里写。 When a node runs, the application layer rebuilds an execution context from the run and hands it to the domain engine. A node reads it for inputs and writes global_context for outputs, which downstream nodes then read. Within it, global_context.__runtime__ is a runtime namespace reserved by the engine — nodes should never write into it directly.

字段Field 类型Type 作用Purpose

每当一个节点执行完成，应用层会把结果写进 __runtime__：累积每个节点的输出（node_outputs），并刷新 last_node_id / last_output / last_status 三个"上一节点"快捷键，供后续节点的模板表达式直接引用。 As each node finishes, the application layer writes the result into __runtime__: it accumulates every node's output (node_outputs) and refreshes the three "last node" shortcuts last_node_id / last_output / last_status, which later nodes' template expressions reference directly.

两台状态机Two state machines

运行有运行的生命周期，节点有节点的生命周期——前者是后者的汇总结果。A run has its own lifecycle and a node has its own — the former is an aggregate of the latter.

运行状态 WorkflowStatusRun status — WorkflowStatus

运行从 PENDING 创建，派发首节点后转 RUNNING，最终落到四种终态之一。END 节点成功即整条运行 SUCCESS；任一节点失败则运行 FAILURE。 A run is created as PENDING, turns RUNNING once the first node is dispatched, and finally settles into one of four terminal states. A successful END node marks the whole run SUCCESS; any failing node marks it FAILURE. 

# 运行状态流转 / Run state transitions
PENDING ──dispatch first node──> RUNNING
                                  │
        ┌─────────────────────────┼─────────────────────────┐
        ▼            ▼             ▼             ▼
     SUCCESS      FAILURE      CANCELLED       PAUSED
   (END ok)   (node fail)  (cancelled)   (paused)
状态State含义Meaning
PENDING运行已创建，尚未派发任何节点。Run created, no node dispatched yet.
RUNNING至少一个节点已被派发，流程进行中。At least one node dispatched; execution in progress.
SUCCESS终态：END 节点以 SUCCESS 收尾。Terminal: an END node finished with SUCCESS.
FAILURE终态：某节点失败，或编排断点（成功节点却无下游）被守护逻辑判定为失败。Terminal: a node failed, or an orchestration dead-end (a successful node with no downstream) was failed by the guard.
CANCELLED终态：流程被取消，后续节点不再执行。Terminal: the run was cancelled; remaining nodes do not run.
PAUSED暂停态，等待人工或外部恢复。Paused, awaiting manual or external resumption.

节点状态 NodeStatusNode status — NodeStatus

节点实例从 PENDING 进入 RUNNING，正常结束为 SUCCESS 或 FAILURE；分支未走到的节点为 SKIPPED；ACTION 异步节点进 SUSPENDED 等回调；带重试的节点在尝试间处于 RETRYING；随流程取消的节点为 CANCELLED。 A node instance moves from PENDING to RUNNING, ending normally as SUCCESS or FAILURE; branches not taken become SKIPPED; async ACTION nodes enter SUSPENDED awaiting a callback; nodes with retries sit in RETRYING between attempts; nodes cancelled with the run become CANCELLED. 

# 节点状态流转 / Node state transitions
PENDING ──> RUNNING ──┬──> SUCCESS
                      ├──> FAILURE
                      ├──> SKIPPED     # 分支未选中 / branch not taken
                      ├──> SUSPENDED ──callback──> SUCCESS   # ACTION async_mode
                      ├──> RETRYING  ──re-attempt──> RUNNING   # attempt_count++
                      └──> CANCELLED   # 跟随流程取消 / cancelled with run

两个特殊节点状态Two special node states

SUSPENDED 与 RETRYING 是引擎"异步等待"和"自愈重试"能力的状态投影。SUSPENDED and RETRYING are the state projections of the engine's "async wait" and "self-healing retry" capabilities.

⏸️

SUSPENDED — 等待外部回调Awaiting external callback

当 ACTION 节点配置 async_mode=true，引擎发出请求后不等结果，把节点置为 SUSPENDED 并在请求头注入 X-Callback-Url。外部系统处理完毕后回调该地址，应用层据此把节点改为 SUCCESS 并派发下游。适合长耗时、异步审批类外部交互。When an ACTION node sets async_mode=true, the engine fires the request but does not wait for the result: it marks the node SUSPENDED and injects X-Callback-Url into the request headers. Once the external system finishes, it calls that URL back; the application layer then flips the node to SUCCESS and dispatches downstream. Ideal for long-running or async-approval interactions.

🔁

RETRYING — 重试中Retrying

节点配置可带重试策略（最大尝试次数、间隔、按响应码重试）。每一次尝试都会让节点实例的 attempt_count 自增，并各写一条 ExecutionLog（attempt_no 递增），便于看清"第几次才成功"。重试窗口内节点处于 RETRYING。A node may carry a retry policy (max attempts, interval, retry-on-status). Each attempt increments the node instance's attempt_count and writes its own ExecutionLog (with an increasing attempt_no), so you can see "which attempt finally succeeded". During the retry window the node sits in RETRYING.

版本与快照：让运行结果可复现Versioning & snapshots: reproducible runs

定义会被持续编辑，但已经在跑的运行必须"看到"触发那一刻的样子。Definitions keep changing, but a running execution must "see" the definition exactly as it was at trigger time.

引擎用两道机制保证这一点： The engine guarantees this with two mechanisms:

  1. 保存即建版本快照Save creates a version snapshot

    创建、更新、回滚工作流定义时，都会向 workflow_definition_version_tbl 追加一条只读快照，并把 version 号自增。每条快照带 change_summary（如 "Initial version" / "Definition updated" / "Rollback to v3"），形成可追溯的演进历史。Creating, updating or rolling back a definition appends a read-only snapshot to workflow_definition_version_tbl and bumps the version number. Each snapshot carries a change_summary (e.g. "Initial version" / "Definition updated" / "Rollback to v3"), forming a traceable evolution history.

  2. 运行绑定定义快照Runs bind a definition snapshot

    触发时，应用层把当前定义整体序列化进运行实例的 definition_snapshot 字段，并记下 definition_version。此后该运行的所有节点都从这份快照取配置——即使有人在运行途中改了或删了定义，正在跑的流程也不受影响。At trigger time the application layer serializes the whole current definition into the run's definition_snapshot field and records definition_version. From then on every node of that run reads its config from this snapshot — so even if someone edits or deletes the definition mid-run, the in-flight execution is unaffected.

  3. 回滚恢复历史版本Rollback restores a historical version

    回滚时，引擎从目标版本快照取出 nodes_config + global_context（连同 name / schedule / graph_data 等）写回定义，并再生成一条新版本记录这次回滚——历史只增不删。整个回滚在分布式锁内进行，避免与并发更新打架。On rollback the engine restores nodes_config + global_context (along with name / schedule / graph_data) from the target version snapshot back into the definition, and creates yet another new version recording the rollback — history only ever grows. The whole rollback runs inside a distributed lock to avoid racing concurrent updates.

设计 / 运行解耦的回报。因为运行携带自己的定义快照，你可以放心地一边运营线上工作流、一边迭代下一版——两者互不打扰。版本号则让"它当时到底跑的是哪一版"永远有答案。 The payoff of decoupling design from runtime. Because a run carries its own definition snapshot, you can safely operate live workflows while iterating the next version — neither disturbs the other. The version number means "which version did it actually run?" always has an answer.
