opsbffsrv — 运维 BFFOps Backend-for-Frontend

为运维前端（adminops UI）聚合接口的 BFF：一侧用两步认证守护前端，另一侧把服务管理请求转发给 infrasrv，并整合服务运维、日志、工作流运维等能力。 The BFF that aggregates APIs for the ops frontend (adminops UI): it guards the frontend with two-step auth on one side, forwards service-management requests to infrasrv on the other, and folds in service ops, logging and workflow ops.

概述Overview

前端的唯一后端入口：聚合、认证、转发。The frontend's single backend entry: aggregate, authenticate, forward.

opsbffsrv 扮演经典 BFF 角色——前端不直接触达 infrasrv/controllersrv，而是统一调 opsbffsrv。它做三件事：(1) 用 session Cookie 守护除登录外的所有接口；(2) 把服务启停、注册概览、实例统计等请求转发给 infrasrv（经 infrasrv_api 客户端）并裁剪/聚合成前端友好的形状；(3) 自持 认证、业务日志检索、工作流运维与 Caddy 网关 / 数据库管理能力。 opsbffsrv plays the classic BFF role — the frontend never touches infrasrv/controllersrv directly, only opsbffsrv. It does three things: (1) guards every endpoint except login with a session cookie; (2) forwards service start/stop, registry overview and instance stats to infrasrv (via the infrasrv_api client) and reshapes the result for the UI; (3) owns auth, business-log search, workflow ops and Caddy gateway / DB admin capabilities.

🛂

认证保护Auth-guarded

auth middleware 校验 ops_session Cookie；仅 /auth/* 与内部健康探针豁免。An auth middleware checks the ops_session cookie; only /auth/* and internal health probes are exempt.

↪️

向 infrasrv 转发Forwards to infrasrv

服务启停、注册、实例与日志类请求经 infrasrv_api 客户端转发，BFF 只做聚合裁剪。Service start/stop, registry, instance and log calls are forwarded via the infrasrv_api client; the BFF only aggregates and trims.

🧰

自持运维能力Owns ops surfaces

工作流运维、业务日志检索、Caddy 网关、数据库管理由 opsbffsrv 直接承载。Workflow ops, business-log search, the Caddy gateway and DB admin are hosted directly by opsbffsrv.

🔐

凭据安全Credential safety

密码用 bcrypt 哈希，TOTP secret 用 Fernet 加密落库。Passwords are bcrypt-hashed; the TOTP secret is Fernet-encrypted at rest.

所有接口前缀为 /v1/opsbffsrv，响应为 APIResponse 信封，下表“关键字段”指 data 内容。 All endpoints are prefixed /v1/opsbffsrv; responses use the APIResponse envelope, and "key fields" below describe data.

认证与授权 APIAuth & Authorization API

两步登录：密码 + 验证码 → TOTP；通过后下发 session Cookie。Two-step login: password + captcha → TOTP, then a session cookie is issued.

方法Method 路径Path 用途Purpose 关键字段Key fields
GET/v1/opsbffsrv/auth/captcha 取图形验证码Get image captcha 响应Resp: captcha_id, image_base64
POST/v1/opsbffsrv/auth/login 第一步：用户名+密码+验证码Step 1: username + password + captcha 请求Req: username, password, captcha_id, captcha_answer · 响应Resp: next_step(totp_bind/totp_verify), temp_token
POST/v1/opsbffsrv/auth/totp/setup 取 TOTP 绑定二维码Get TOTP provisioning QR 请求Req: temp_token · 响应Resp: otpauth_uri, qr_code_base64
POST/v1/opsbffsrv/auth/totp/bind 第二步(首次)：完成 TOTP 绑定Step 2 (first time): bind TOTP 请求Req: temp_token, totp_code · 响应Resp: 设置 ops_session Cookie
POST/v1/opsbffsrv/auth/totp/verify 第二步(常规)：TOTP 验证Step 2 (regular): verify TOTP 请求Req: temp_token, totp_code · 响应Resp: 设置 ops_session Cookie
GET/v1/opsbffsrv/auth/session 校验当前 sessionCheck current session 请求Req: ops_session Cookie · 响应Resp: authenticated, username

登录是两阶段：login 校验密码（bcrypt）与图形验证码，并按 IP 锁定防暴破，成功后返回短时 temp_token 与 next_step。未绑过 TOTP 走 totp/setup + totp/bind；已绑过则直接 totp/verify。两者成功都把会话 token 写入 HttpOnly 的 ops_session Cookie（SameSite=lax，可配 Secure）。TOTP secret 以 Fernet 加密存储。 Login is two-phase: login validates the password (bcrypt) and image captcha with per-IP lockout against brute force, then returns a short-lived temp_token and a next_step. First-time users go through totp/setup + totp/bind; returning users go straight to totp/verify. Either success writes the session token into an HttpOnly ops_session cookie (SameSite=lax, optional Secure). The TOTP secret is stored Fernet-encrypted.
