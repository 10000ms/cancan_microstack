安装Install

一行 pip 命令装好整个工具箱，附带运行所需的姊妹库。One pip command installs the whole toolkit, along with the sibling libraries it needs.

# 安装 / Install
pip install cancan-microstack

这会一并带上 linglong-web[all]（Web 服务框架，配置 PostgreSQL / Redis / MongoDB / RabbitMQ / Celery）与 dragonfly-container[all]（容器操作封装）。 This pulls in linglong-web[all] (the web service framework that wires up PostgreSQL / Redis / MongoDB / RabbitMQ / Celery) and dragonfly-container[all] (the container-control wrapper).

前置条件Prerequisites

要求Requirement说明Detail
Python ≥ 3.10库本身的运行环境。Runtime for the library itself.
Docker 或 or Podman运行容器栈所需；两者皆可，CLI 会自动探测。Required to run the container stack; either works, the CLI auto-detects.
两种调用方式。安装后即提供 cancan 命令；若 PATH 未生效，也可用 python -m cancan_microstack.cli，两者等价。 Two ways to invoke. Installation provides the cancan command; if it is not on your PATH, python -m cancan_microstack.cli is exactly equivalent.

浏览内置资产Browse the bundled assets

先看库里随包发布了哪些 compose / DDL / 脚本 / 运维前端，再决定起什么。First see what compose / DDL / scripts / ops UI ship with the library, then decide what to bring up.

# 列出全部资产 / List every asset
cancan assets list

# 只看 docker 子目录 / Filter to the docker subdir
cancan assets list docker

每条记录给出资产的逻辑名（如 docker/docker-compose.infra.yml）、物化后的路径与是否为目录。逻辑名是对外稳定的引用方式，无需关心库内部文件实际落在哪里。 Each record shows the asset's logical name (e.g. docker/docker-compose.infra.yml), its materialized path, and whether it is a directory. The logical name is the stable reference — you never need to know where the file physically lives inside the package.

脚手架与预检Scaffold & pre-flight

新工作区先 init 生成最小骨架，再 doctor 体检。For a fresh workspace, init generates a minimal skeleton, then doctor checks readiness.

# 生成业务 compose 样例 + .env（含自动生成的 TOTP key）+ overrides 骨架
# Scaffold a services compose sample + .env (generated TOTP key) + overrides skeleton
cancan init

# 起栈前体检：容器引擎 / 端口 / 配置 / 弱默认值
# Pre-flight: engine / ports / config / weak defaults
cancan doctor

doctor 会提示仍在使用的弱默认口令——生产前在 .env 改掉。doctor flags weak default secrets still in use — change them in .env before production.

一键起栈Bring up the whole stack

一条 stack up 同时管理宿主机进程与容器栈，省去手动拼装。A single stack up manages both the host process and the container stack — no manual wiring.

# 在业务仓根目录执行 / Run from your product repo root
cancan stack up --workspace . --service-file docker-compose.services.yml

这条命令按顺序做四件事：This command does four things in order:

  1. 合成 compose 栈文件Synthesize the compose stack file

    以内置 infra compose 为基底，深度合并你的业务 docker-compose.services.yml 与任意 override，输出 compose.cancan.yml。Takes the built-in infra compose as the base, deep-merges your business docker-compose.services.yml and any overrides, and writes compose.cancan.yml.

  2. bootstrap 工作区运行文件Bootstrap workspace run files

    缺失才生成：cmd/*/run.py 服务包装器、用于 include 的 infra compose、镜像脚手架与 DDL、以及供 Caddy 挂载的 adminops 静态前端。已有文件不会被覆盖。Generated only when missing: cmd/*/run.py service wrappers, the include-ready infra compose, image scaffolds and DDL, and the adminops static frontend for Caddy to serve. Existing files are never overwritten.

  3. 宿主机后台起 controllersrvStart controllersrv on the host

    在宿主机以守护进程方式启动 controllersrv，用 pidfile 管理（server_log_data/pids/controllersrv.pid），随后轮询 /internal/health 等待就绪。Launches controllersrv as a host daemon, tracked by a pidfile (server_log_data/pids/controllersrv.pid), then polls /internal/health until it is ready.

  4. 拉起容器栈Bring the container stack up

    执行 docker compose up -d（或 podman compose）让其余容器后台运行；日志写入 server_log_data/stack/compose.up.log。Runs docker compose up -d (or podman compose) to bring the remaining containers up in the background; logs go to server_log_data/stack/compose.up.log.

为什么是宿主机 + 容器两层？controllersrv 要操作 Docker/Podman，不能由 compose 自己管理，因此它跑在宿主机；其余服务是容器。stack up 把这两层一起拉起，所以你只需一条命令。 Why host plus containers? controllersrv drives Docker/Podman and therefore cannot be managed by compose itself, so it runs on the host; everything else is a container. stack up brings both layers up together, so you only run one command.

常用开关：--engine docker|podman|auto 指定容器引擎（默认 auto）；--build 强制重建镜像；--no-with-controllersrv 跳过宿主机进程；--no-bootstrap 跳过运行文件生成。完整参数见 CLI 参考。 Handy flags: --engine docker|podman|auto picks the container engine (default auto); --build forces an image rebuild; --no-with-controllersrv skips the host process; --no-bootstrap skips run-file generation. See the CLI Reference for the full set.

查看状态与关闭Check status and tear down

同一套命令观察与回收整栈，包括宿主机进程与容器。The same command family observes and reclaims the whole stack — host process and containers alike.

# 查看状态：controllersrv pid + 容器 compose ps
# Status: controllersrv pid + container compose ps
cancan stack status --workspace .

# 关闭整栈（容器 down + 停止 controllersrv）
# Tear down (containers down + stop controllersrv)
cancan stack down --workspace .

# 连带清理命名数据卷（会丢数据）/ Also drop named volumes (destroys data)
cancan stack down --workspace . --volumes
--volumes 会删除命名数据卷，PostgreSQL / MongoDB 等持久化数据将一并丢失。仅在确实要清空环境时使用。 --volumes removes named volumes, so persisted data in PostgreSQL / MongoDB and friends is lost with it. Use it only when you genuinely want a clean slate.

生产安全Production safety

开箱默认值是为开发便利，不是为生产准备的。The out-of-the-box defaults are for convenience, not for production.

切勿沿用默认凭证。内置 compose 与服务配置使用本地开发的弱默认值（例如 admin123）以便开箱即用，这些值均可被环境变量覆盖。生产部署时，请务必通过环境变量注入真实凭证，绝不要把默认值带上线。 Never ship the default credentials. The bundled compose and service configs use weak local-dev defaults (e.g. admin123) so the stack works out of the box, and every one of them is overridable via environment variables. In production, always inject real secrets through env vars — never carry the defaults to a live deployment.
