总体架构Architecture

一个很薄的门面，下面挂着三个子系统、一层运行时策略，以及三个内置服务——这就是 Cancan 的全部骨架。理解了它，库里其它代码都能对号入座。 A very thin facade sitting on top of three subsystems, one runtime-strategy layer and three bundled services — that is the entire skeleton of Cancan. Once you have it, every other module finds its place.

总体架构Big picture

门面在上，能力居中，运行时策略落地，服务与数据在下。Facade on top, capabilities in the middle, runtime strategy beneath, services and data at the bottom.

最外层是 CancanMicrostack 门面（core/microstack.py）。它只暴露三个高层方法，把活儿分给三个子系统：AssetManager 管包内资产，ComposeBuilder 合成 compose，ServiceRunner 启动服务。三者之下是 runtime/ 这一层运行时策略，决定“库被嵌进某个业务仓后，到底在哪儿跑、用谁的文件”。最底下才是真正干活的三个内置服务和它们依赖的数据层。 The outermost layer is the CancanMicrostack facade (core/microstack.py). It exposes only three high-level methods and delegates the work to three subsystems: AssetManager for packaged assets, ComposeBuilder for compose synthesis, ServiceRunner for starting services. Below them sits the runtime/ strategy layer, which decides — once the library is embedded into a host repo — where it runs and whose files it uses. The three bundled services and their data stores live at the very bottom.

The facade delegates to the subsystems; at runtime controllersrv runs on the host (it drives the container engine) while infrasrv / opsbffsrv run inside containers.

core 核心能力Core capabilities

资产、合成、运行——三件事，三个类。Assets, synthesis, running — three jobs, three classes.

AssetManager — 把静态文件当包资源assets as package resources

资产指 cancan_microstack/assets/ 里随包发布的静态文件：compose 模板、DDL、脚本、运维前端等。AssetManager 用标准库 importlib.resources 访问它们——resources.files(...) 拿到可遍历的资源树，resources.as_file(...) 把资源“物化”成真实路径，即使资源被打进 wheel/zip 也能用。 An asset is a static file shipped with the package under cancan_microstack/assets/: compose templates, DDL, scripts, the ops UI. AssetManager reaches them through the stdlib importlib.resources — resources.files(...) gives a traversable tree, and resources.as_file(...) materializes a resource into a real path, even when it is bundled inside a wheel/zip.

  • 逻辑名（logical name）：调用方用稳定的逻辑路径引用资产，例如 docker/docker-compose.infra.yml，不必关心库内部真实文件在哪。Logical name: callers reference assets by a stable path such as docker/docker-compose.infra.yml, never the library's real on-disk location.
  • list_assets(subdir) 递归遍历并返回 AssetRecord(logical_name, path, is_dir)；遍历时跳过 __pycache__ 与 .DS_Store。list_assets(subdir) walks recursively and returns AssetRecord(logical_name, path, is_dir), skipping __pycache__ and .DS_Store.
  • export_asset(...) 把资产复制进工作区：目录用 shutil.copytree，文件用 shutil.copy2。这一步很关键——Docker 只看挂载进来的工作区路径，不会去 venv 里翻 site-packages。export_asset(...) copies an asset into the workspace: shutil.copytree for directories, shutil.copy2 for files. This matters — Docker only sees the mounted workspace path, never site-packages inside the venv.
  • read_text(...) 读文本，resolve_path(...) 物化出一个可读的 Path（ComposeBuilder 就用它加载 base compose）。read_text(...) reads text; resolve_path(...) materializes a readable Path (used by ComposeBuilder to load the base compose).
一个易踩的坑：不要用 as_file(...) 物化出的路径反推逻辑名。打包成 zip/wheel 时每个节点可能被物化到不同临时目录，relative_to() 会失败并丢前缀（返回 infra 而不是 ddl/infra）。源码因此在遍历时自行拼接 logical_name。 A real trap: never derive the logical name from a path materialized by as_file(...). When packaged as a zip/wheel, each node may land in a different temp dir, so relative_to() fails and drops the prefix (returning infra instead of ddl/infra). The code therefore builds logical_name itself during the walk.

ComposeBuilder — 把多份 compose 深度合并成一个栈deep-merge many composes into one stack

合成的顺序固定：以内置 infra compose 为 base（默认资产 docker/docker-compose.infra.yml），先合并业务 compose（可选），再依次合并多份 overrides，最后写出 compose.cancan.yml（默认在工作区根）。后应用的优先级更高。 The synthesis order is fixed: start from the built-in infra compose as the base (default asset docker/docker-compose.infra.yml), merge the business compose (optional), then merge the overrides in order, and finally write compose.cancan.yml (workspace root by default). Whatever is applied later wins. 

# 合成顺序 / merge order
base (docker/docker-compose.infra.yml)
  → deep_merge service_file   # 业务 compose，可选 / optional
  → deep_merge override[0]    # 多份覆盖，依次叠加 / overrides, in order
  → deep_merge override[1]
  → compose.cancan.yml      # yaml.safe_dump(sort_keys=False)

递归合并规则 _deep_merge(base, override) 只有三条：The recursive rule _deep_merge(base, override) has just three cases:

  1. key 在 base 里不存在 → 直接取 override 的值（深拷贝）。key absent in base → take the override value (deep-copied).
  2. 两边都是 dict → 递归合并。both sides are dicts → recurse.
  3. 其它情况 → override 覆盖 base（包括列表，整段替换而非拼接）。otherwise → override replaces base (lists included — replaced wholesale, not concatenated).

写出前还有两处清洗：移除已弃用的 version 字段；对非 external 的网络去掉固定 name，让 compose 生成项目级网络名以避免与历史网络撞名；并对带 build 的服务补 pull_policy: never，优先本地构建。Before writing it also normalizes twice: it drops the deprecated version field; for non-external networks it removes the fixed name so compose generates a project-scoped name (avoiding collisions with legacy networks); and for services with a build section it sets pull_policy: never to prefer local builds.

ServiceRunner — 服务名 → 入口点字符串 → 动态调用name → entrypoint string → dynamic call

ServiceRunner 把服务名映射成一个“入口点字符串”，再动态 import 并调用其 main(host, port)。映射写死在 _ENTRYPOINTS 字典里： ServiceRunner maps a service name to an "entrypoint string", then dynamically imports it and calls its main(host, port). The mapping is a hard-coded _ENTRYPOINTS dict: 

controllersrv -> cancan_microstack.cmd.controllersrv.run:main
infrasrv      -> cancan_microstack.cmd.infrasrv.run:main
opsbffsrv     -> cancan_microstack.cmd.opsbffsrv.run:main
  • run(...) 是同步入口，内部 asyncio.run(run_async(...))。run(...) is the sync entry; it wraps asyncio.run(run_async(...)).
  • 启动前先 bootstrap_from_workspace(...)，把工作区与覆盖目录一次性配好。Before launch it runs bootstrap_from_workspace(...) to wire up the workspace and override dir in one shot.
  • 用 inspect.isawaitable(result) 兼容同步/异步入口：是 awaitable 就 await，否则直接返回。所以 cmd/* 的 main 写成同步或异步都行。It uses inspect.isawaitable(result) to support both sync and async entries: if awaitable it awaits, otherwise it returns. So a cmd/* main can be written either way.

整栈生命周期则由 StackManager（core/stack_manager.py）统一管理：它既调 ComposeBuilder 出文件、调 compose 引擎起容器，也负责把 controllersrv 作为宿主机 daemon 后台拉起（pidfile 管理）并做健康检查。 Whole-stack lifecycle is owned by StackManager (core/stack_manager.py): it builds the compose file, drives the compose engine to bring up containers, and also launches controllersrv as a background host daemon (pidfile-managed) and health-checks it.

runtime 运行时策略Runtime strategy

库被嵌进业务仓后，“在哪儿跑、用谁的文件、怎么后台运行”都在这一层。Once embedded into a host repo, "where it runs, whose files it uses, how it daemonizes" all live here.

模块Module 职责Responsibility 关键点Key points
workspace.py 探测并配置工作区根目录。Detect and configure the workspace root. 优先环境变量 CANCAN_WORKSPACE_ROOT；否则从 cwd 向上找标记（cmd / pyproject.toml / server_log_data 等）。提供 ensure_server_log_dir / ensure_subdir。Prefers env CANCAN_WORKSPACE_ROOT; otherwise walks up from cwd for markers (cmd / pyproject.toml / server_log_data …). Offers ensure_server_log_dir / ensure_subdir.
overrides.py 不 fork 就能局部覆盖服务实现/配置。Override service code/config locally without forking. 发现 cancan_overrides/（或 CANCAN_OVERRIDE_ROOT），用 pkgutil.extend_path 把 cancan_overrides/<service> 插到搜索路径最前。bootstrap_from_workspace 把它和 workspace 串起来。Discovers cancan_overrides/ (or CANCAN_OVERRIDE_ROOT) and prepends cancan_overrides/<service> to the search path via pkgutil.extend_path. bootstrap_from_workspace ties it to the workspace.

一条完整调用链A full call chain

以 cancan services run infrasrv 为例，看一次命令如何抵达服务入口。Tracing cancan services run infrasrv from the CLI down to the service entry.

  1. CLI 解析：cli/main.py:main() 用 argparse 解出 name=infrasrv、host、port、workspace，并构造 stack = CancanMicrostack()。CLI parse: cli/main.py:main() parses name=infrasrv, host, port, workspace via argparse and constructs stack = CancanMicrostack().
  2. 委托门面：调用 stack.run_service("infrasrv", host, port, workspace)，门面记一行日志后转交 ServiceRunner。Delegate to facade: calls stack.run_service("infrasrv", host, port, workspace); the facade logs once and hands off to ServiceRunner.
  3. 进事件循环：ServiceRunner.run(...) 执行 asyncio.run(run_async(...))。Enter the loop: ServiceRunner.run(...) executes asyncio.run(run_async(...)).
  4. bootstrap：bootstrap_from_workspace(workspace) 先 configure_workspace，再发现并 configure_overrides。bootstrap: bootstrap_from_workspace(workspace) first configure_workspace, then discovers and configure_overrides.
  5. 解析入口点：从 _ENTRYPOINTS 取出 cancan_microstack.cmd.infrasrv.run:main，import_module + getattr 拿到函数。Resolve entrypoint: look up cancan_microstack.cmd.infrasrv.run:main in _ENTRYPOINTS, then import_module + getattr.
  6. 调用 main：result = main(host=host, port=port)；若 inspect.isawaitable(result) 则 await。服务入口从这里接管 web server、路由与配置。Invoke main: result = main(host=host, port=port); if inspect.isawaitable(result) then await. The service entry now takes over the web server, routes and config.

整栈启动流程Stack startup

cancan stack up 把构建、引导、daemon、容器、健康检查串成一条流水线。cancan stack up chains build, bootstrap, daemon, containers and health check into one pipeline.

  1. 构建 compose：StackManager.build(options) → ComposeBuilder 合成 compose.cancan.yml。Build compose: StackManager.build(options) → ComposeBuilder synthesizes compose.cancan.yml.
  2. bootstrap 工作区：缺失则生成 cmd/* 包装器、导出 infra compose / DDL / 运维前端等资产（_bootstrap_workspace_files，不覆盖已有文件）。Bootstrap workspace: generate cmd/* wrappers and export infra compose / DDL / ops UI assets when missing (_bootstrap_workspace_files, never overwriting existing files).
  3. 起 controllersrv daemon：start_controllersrv 以宿主机后台进程拉起 cancan services run controllersrv，写 pidfile，并把容器引擎偏好透传给它。Start controllersrv daemon: start_controllersrv launches cancan services run controllersrv as a background host process, writes a pidfile and propagates the container-engine preference.
  4. compose up -d：compose_up 经 run_compose_streaming 拉起 infrasrv / opsbffsrv 等容器，日志同时落到 server_log_data/stack/compose.up.log。compose up -d: compose_up via run_compose_streaming brings up containers (infrasrv / opsbffsrv …), teeing logs to server_log_data/stack/compose.up.log.
  5. 健康检查：wait_controllersrv_ready 在超时窗口内轮询 /internal/health，未就绪则打印警告并提示查看 daemon 日志。Health check: wait_controllersrv_ready polls /internal/health within a timeout; if not ready it warns and points to the daemon log.

设计原则Design principles

整套架构其实只立足于两个想法。The whole architecture rests on just two ideas.

📦

资产即包资源Assets as package resources

compose / DDL / 脚本 / 运维前端跟代码一起发布，由 AssetManager 统一“列出 / 导出 / 读取 / 解析路径”。基础设施随库版本演进，业务仓不再 copy-paste 模板。compose / DDL / scripts / ops UI ship with the code and are listed / exported / read / resolved through AssetManager. Infrastructure evolves with the library version — no more copy-pasted templates in product repos.

🧭

工作区 + 覆盖作为运行时策略统一注入Workspace + overrides injected as one runtime strategy

workspace 决定“你在哪个业务仓里跑”，overrides 让业务仓用 cancan_overrides/ 局部替换实现，ServiceRunner 启动前统一 bootstrap——一致性内建，服务自身不必各写一套路径探测。workspace decides which host repo you run in, overrides lets the repo swap pieces via cancan_overrides/, and ServiceRunner bootstraps both before launch — consistency is built in, so services don't each reinvent path detection.

一句话收尾：core 提供能力，runtime 提供“在业务仓里怎么落地”的策略，CLI 把能力串成统一工具链。 In one line: core provides capabilities, runtime provides the strategy for landing them in a host repo, and the CLI threads the capabilities into one toolchain.
