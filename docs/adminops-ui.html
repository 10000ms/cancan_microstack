cancan 只包含前端构建产物Cancan ships only the build artifact

前端源码在独立私有仓维护；本库随包发布的是已经 vite build 出来的静态 dist。The frontend source lives in a separate private repo; what this library ships is the already-built static dist from vite build.

运维前端是一个 Vite 构建的单页应用（SPA），随库发布在资产路径 assets/www/adminops/ 下：一个 index.html、一份资源清单，加上按视图切分的 JS / CSS chunk（如 ServiceList-*.js、WorkflowDesigner-*.js，文件名带内容哈希）。本库不含前端源码，只是把它的产物当作一项资产携带。 The ops UI is a Vite-built single-page application (SPA) shipped under the asset path assets/www/adminops/: one index.html, an asset manifest, and per-view JS / CSS chunks (e.g. ServiceList-*.js, WorkflowDesigner-*.js, with content-hashed filenames). This library carries no frontend source — only the artifact, treated as an asset.

更新流程（三步）。① 在私有源码仓里 build；② 把产出的 dist/ 整体覆盖到本库的 assets/www/adminops/；③ 重新发布 cancan 库版本。运维前端因此随库版本演进——和 compose / DDL 等其他资产一样，由库统一分发，业务仓不必自行维护或拷贝。 Update flow (three steps). ① build in the private source repo; ② copy the resulting dist/ over this library's assets/www/adminops/; ③ cut a new cancan release. The ops UI thus evolves with the library version — like compose / DDL and the other assets, distributed by the library so product repos never maintain or copy it.

如何被托管How it is served

起栈时物化到工作区，Caddy 作为静态 + SPA 回退托管，API 反代到 opsbffsrv。Materialized into the workspace at stack-up, served by Caddy as static + SPA fallback, with the API reverse-proxied to opsbffsrv.

cancan stack up 的 bootstrap 阶段会把这份前端导出到工作区（默认 builds/caddy/www/adminops，已存在则不覆盖）。运行时 Caddy 把它挂在 /adminops/：静态资源 /adminops/assets/* 直出，其余路径回退到 index.html 以支持前端路由；index.html 带 no-store 缓存头，避免发布后 chunk 不匹配。前端调用的所有接口（/v1/opsbffsrv/*）由 Caddy 反向代理到 opsbffsrv。 During the bootstrap phase of cancan stack up, this frontend is exported into the workspace (by default builds/caddy/www/adminops, never overwriting an existing copy). At runtime Caddy mounts it at /adminops/: static assets under /adminops/assets/* are served directly, while every other path falls back to index.html for client-side routing; index.html carries a no-store cache header to avoid chunk mismatches after a release. Every API the frontend calls (/v1/opsbffsrv/*) is reverse-proxied by Caddy to opsbffsrv.

核心运维流程Core ops flows

三条最常用链路，展示前端如何串起 opsbffsrv 接口。Three everyday flows showing how the frontend strings opsbffsrv endpoints together.

① 服务启停① Start / stop a service

  1. 在 ServiceCard 上发起动作Trigger from a ServiceCard

    点击实例卡片的启 / 停 / 重启，前端 POST async/service/start（或 stop / restart）。这是异步接口，立即返回一个操作 ID。Clicking start / stop / restart on an instance card calls POST async/service/start (or stop / restart). It is an async endpoint that returns an operation ID immediately.

  2. 弹出进度对话框并轮询Open the progress dialog and poll

    OperationProgressDialog 用返回的操作 ID 反复 GET operation/status，把进度回显给用户，直到状态终态（成功 / 失败）。OperationProgressDialog repeatedly calls GET operation/status with that operation ID, echoing progress until it reaches a terminal state (success / failure).

  3. 刷新列表Refresh the list

    完成后回到 ServiceList / ServiceDetail，重新拉取实例与状态。On completion the user returns to ServiceList / ServiceDetail, which re-fetch instances and status.

② 工作流编排② Orchestrate a workflow

  1. 在工作流中心建 / 选工作流Create or pick a workflow in the center

    WorkflowCenter 列出全部工作流，进入某个工作流的设计器。WorkflowCenter lists every workflow; open one in the designer.

  2. 在设计器编排并保存Author and save in the designer

    WorkflowDesigner 画好节点图，POST/PUT ops/workflows 保存，必要时 POST …/validate 校验、POST …/publish 发布。WorkflowDesigner lays out the node graph, saves via POST/PUT ops/workflows, optionally POST …/validate to check and POST …/publish to publish.

  3. 触发并在运行页看实时图Trigger and watch the live graph

    POST ops/workflows/{id}/trigger 触发一次运行，WorkflowRuns 用 GET ops/runs/{run_id}/graph 渲染实时运行图，点节点看 …/nodes/{node_id}/logs。POST ops/workflows/{id}/trigger starts a run; WorkflowRuns renders the live graph from GET ops/runs/{run_id}/graph, drilling into …/nodes/{node_id}/logs per node.

③ Caddy 路由配置③ Configure a Caddy route

  1. 在路由管理里增改路由Add / edit a route in RouteManage

    RouteManage 通过 POST/PUT caddy/routes 写库——此时只改了配置数据库，尚未下发给 Caddy。RouteManage writes via POST/PUT caddy/routes — this only updates the config database, not Caddy itself yet.

  2. 批量同步到 CaddySync to Caddy in one batch

    POST caddy/routes/sync 把所有已启用路由编译成 Caddy JSON 并经 Admin API 一次性下发，配置才真正生效。POST caddy/routes/sync compiles every enabled route into Caddy JSON and pushes it through the Admin API in one shot — only then does it take effect.

  3. 在统计概览验证流量Verify traffic in StatsOverview

    在 StatsOverview 看新路由的实时流量与趋势，确认转发正常。Confirm the forwarding works by watching the new route's realtime traffic and trend in StatsOverview.
