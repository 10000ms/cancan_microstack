The ops frontend uses a mirror set on opsbffsrv, explained at the end of this page.

定义管理Definition management

工作流定义的增删改查、版本历史、回滚与触发——设计态的全部操作都在这一组。CRUD, version history, rollback and trigger for workflow definitions — every design-side operation lives here.

路由顺序的讲究。/workflows/stats 和 /workflows/alerts 这类固定子路径，在路由表里排在 /workflows/{workflow_id} 之前，以免被当成 workflow_id 误匹配。这是阅读路由表时容易忽略的细节。A note on route order. Fixed sub-paths like /workflows/stats and /workflows/alerts are registered before /workflows/{workflow_id} so they are not mistaken for a workflow_id — an easy detail to miss when reading the route table.

运行查询Run queries

运行态只读：列出运行、查运行图谱状态、查单节点的执行历史。Read-only runtime: list runs, fetch the run graph status, and a single node's execution history.

方法Method 路径Path 用途Purpose 关键字段Key fields
GET/v1/infrasrv/runs分页 / 过滤列出运行实例List runs with paging / filtersworkflow_id · reqid · status · date_from · date_to
GET/v1/infrasrv/runs/{run_id}/status查运行图谱（节点 + 连线 + 各节点状态 / 最新输出）Run graph (nodes + edges + per-node status / last output)run_id → nodes[] · edges[] · status
GET/v1/infrasrv/runs/{run_id}/nodes/{node_id}/history查单节点执行历史（含每次尝试的执行日志）A node's execution history (incl. each attempt's log)run_id · node_id → histories[]

/runs/{run_id}/status 返回的是给可视化用的图谱：每个节点带 status、loop_index、last_output 与画布坐标，前端据此把运行实时状态叠加在工作流图上。它的定义来源是运行绑定的快照，因此即便定义被改也能还原"当时的图"。 /runs/{run_id}/status returns a graph for visualization: each node carries status, loop_index, last_output and canvas coordinates, letting the frontend overlay live run state on the workflow graph. Its definition comes from the run-bound snapshot, so it can reconstruct "the graph as it was" even after the definition changes.

回调Callbacks

异步 ACTION 节点的恢复入口：外部系统办完事，回调这里把挂起节点唤醒。The resume entry for async ACTION nodes: once the external system is done, it calls back here to wake the suspended node.

方法Method 路径Path 用途Purpose 关键字段Key fields
POST/v1/infrasrv/callbacks/{node_instance_id}异步 ACTION 完成回调，恢复运行并派发下游Async ACTION completion callback; resume the run and dispatch downstreamnode_instance_id · 回调载荷callback payload → status (ACCEPTED / IGNORED)
该地址正是 ACTION 节点 async_mode 触发时注入到请求头 X-Callback-Url 的值（由 infrasrv host 拼接而成）。只有当对应节点实例确实处于 SUSPENDED 时回调才被接受（ACCEPTED）；否则忽略（IGNORED），从而幂等地抵御重复回调。 This is exactly the value injected into the X-Callback-Url request header when an ACTION node's async_mode fires (built from the infrasrv host). The callback is accepted (ACCEPTED) only when the target node instance is genuinely SUSPENDED; otherwise it is ignored (IGNORED), idempotently shrugging off duplicate callbacks.

统计与告警Statistics & alerts

引擎的运维可观测面：整体统计指标，以及编排守护逻辑产出的引擎告警与处置接口。The engine's ops observability surface: overall statistics, plus engine alerts emitted by the orchestration guard and their handling endpoints.

方法Method 路径Path 用途Purpose 关键字段Key fields
GET/v1/infrasrv/workflows/stats工作流总体统计Overall workflow statisticstotal_runs · runs_by_status · success_rate_24h · next_cron_window
GET/v1/infrasrv/workflows/alerts列出引擎告警（可按状态 / 严重度 / 运行过滤）List engine alerts (filter by status / severity / run)status · severity · run_id
POST/v1/infrasrv/workflows/alerts/{alert_id}/ack标记告警为已知晓Acknowledge an alertoperator · note
POST/v1/infrasrv/workflows/alerts/{alert_id}/resolve标记告警为已解决Resolve an alertoperator · note

引擎告警由编排守护逻辑在异常时落库，便于运维看清"哪条运行的哪个节点出了什么问题"。常见原因包括：运行 ID 非法、运行 / 定义缺失、节点配置缺失、节点执行异常、以及成功节点却无下游的编排断点。告警有 OPEN → ACKED → RESOLVED 的生命周期与严重程度 / 分类，供前端筛选。 Engine alerts are persisted by the orchestration guard on anomalies, so operators can see "which node of which run failed and why". Common reasons include: invalid run ID, missing run / definition, missing node config, node execution exception, and the orchestration dead-end of a successful node with no downstream. Alerts have an OPEN → ACKED → RESOLVED lifecycle plus severity / category for frontend filtering.

运维镜像接口Ops mirror endpoints

运维前端不直接打 infrasrv，而是经 opsbffsrv 的一套面向 UI 的镜像接口。The ops frontend does not call infrasrv directly; it goes through a UI-oriented mirror set on opsbffsrv.

两套接口，各司其职。上面列的 /v1/infrasrv/… 是引擎自身的能力面。运维前端（adminops UI）实际调用的是 opsbffsrv 上的镜像接口 /v1/opsbffsrv/ops/workflows…——它在透传引擎能力的同时，补充了几个面向运维操作的动作：toggle（启停定义）、duplicate（复制工作流）、publish（发布）、validate（校验）。这套镜像同样覆盖了 stats、alerts（含 ack / resolve）、versions、rollback、trigger 等。两者关系与边界详见 opsbffsrv 服务页与运维前端页。 Two API sets, distinct roles. The /v1/infrasrv/… endpoints listed above are the engine's own capability surface. The ops frontend (adminops UI) actually calls the mirror set on opsbffsrv at /v1/opsbffsrv/ops/workflows… — which passes the engine's capabilities through while adding a few operations-oriented actions: toggle (enable/disable a definition), duplicate (copy a workflow), publish, and validate. This mirror also covers stats, alerts (incl. ack / resolve), versions, rollback and trigger. For the relationship and boundaries, see the opsbffsrv service page and the Admin Ops UI page.
