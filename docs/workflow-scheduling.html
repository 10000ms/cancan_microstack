工作流引擎 · 调度与编排Workflow Engine · Scheduling & Orchestration

概念与节点讲的是"工作流长什么样"，这一章讲"它怎么被触发、怎么一节点接一节点地跑起来"：四种触发方式、Celery 分布式编排与内联降级、贯穿全程的模板表达式，以及保护定义改动的分布式锁。 Concepts and nodes describe "what a workflow looks like"; this chapter describes "how it gets triggered and runs node after node": four ways to trigger, Celery distributed orchestration with an inline fallback, the template expressions woven throughout, and the distributed lock that guards definition changes.

触发方式How runs are triggered

三种触发类型加一种异步恢复机制，覆盖手动、定时、外部与回调四类场景。Three trigger types plus an async-resume mechanism cover manual, scheduled, external and callback scenarios.

运行实例的 trigger_type 记录它从何而来，枚举为 MANUAL / SCHEDULE / API。回调不是一种触发类型——它是用来恢复已挂起节点的独立入口，但同样属于"流程怎么往前走"的一环，故并列于下。 A run's trigger_type records where it came from, enumerated as MANUAL / SCHEDULE / API. A callback is not a trigger type — it is a separate entry point used to resume a suspended node — but since it is part of "how the flow moves forward", it is listed alongside.

触发后应用层做三件事：把当前定义序列化进运行的 definition_snapshot、规范化 trigger_context（补 reqid）、找到 START 节点把首批下游派发出去。若 START 没有下游，运行停在 PENDING，触发响应里的 dispatch_status 会是 QUEUED_NO_ENTRY；正常派发则为 DISPATCHED。 On trigger the application layer does three things: serialize the current definition into the run's definition_snapshot, normalize the trigger_context (filling in reqid), and locate the START node to dispatch the first batch. If START has no downstream, the run stays PENDING and the trigger response's dispatch_status is QUEUED_NO_ENTRY; a normal dispatch yields DISPATCHED.

任务队列与降级：Celery 在线走队列，离线走内联Queue & fallback: Celery when online, inline when not

引擎按"一节点一任务"推进，每派发一个下游就投递一次。它会先看 Celery worker 在不在，再决定走分布式还是内联。The engine advances "one node, one task", enqueuing once per downstream dispatch. It first checks whether a Celery worker is alive, then chooses distributed or inline.

编排的核心是 enqueue_node_execution(run_id, node_id, loop_index)：节点执行完算出下游节点后，对每个下游调一次它。这个函数不直接执行节点，而是决定派发路径： The heart of orchestration is enqueue_node_execution(run_id, node_id, loop_index): after a node finishes and its downstream is computed, it is called once per downstream. This function does not run the node directly — it chooses the dispatch path:

  1. Celery 未初始化 → 内联Celery not initialized → inline

    若进程内没有 Celery 应用，直接走内联执行器。If there is no Celery app in the process, run via the inline executor.

  2. 无活跃 Worker → 内联No active worker → inline

    通过 control.ping（结果缓存约 15 秒）探测 worker 是否存活；无 worker 就降级内联，避免任务投出去无人消费、无限排队。A control.ping (cached ~15s) probes for a live worker; with none, it falls back to inline, so tasks are never enqueued to a queue nobody drains.

  3. Worker 在线 → Celery 队列Worker online → Celery queue

    有 worker 时 send_task 投递 workflows.execute_node 任务，由 execute_node_task 在 worker 侧执行；投递失败也兜底回内联。With a worker, send_task enqueues the workflows.execute_node task, executed worker-side by execute_node_task; a send failure also falls back to inline.

内联降级执行器The inline fallback executor

_InlineFallbackExecutor 是降级路径的关键——它不是"在当前协程里直接 await 节点"，而是一个受限并发的线程内队列池：按事件循环维护一个 asyncio.Queue，启动固定数量的后台 worker 协程串行消费。这样即便 Celery 不可用，瞬时大量分支（如 FORK 一拆十）也不会无限并发把进程压垮；队列满时施加背压。并发数与队列上限由配置决定，支持热更新。 _InlineFallbackExecutor is the key to the fallback path — it does not "await the node directly in the current coroutine" but is a bounded-concurrency in-thread queue pool: it keeps one asyncio.Queue per event loop and starts a fixed number of background worker coroutines to drain it. So even when Celery is unavailable, a sudden fan-out (e.g. a FORK into ten) won't spawn unbounded concurrency and crush the process; backpressure kicks in when the queue fills. Concurrency and queue limits are config-driven and hot-reloadable.

register_inline_orchestrator：打破循环导入register_inline_orchestrator: breaking the import cycle

队列模块（workflow_queue）需要调用应用层的节点编排函数，而应用层（workflow_app）又要 import 队列模块——这是个循环依赖。解法是运行时注册：应用层在模块加载末尾调用 register_inline_orchestrator(workflow_app._execute_node_orchestrator) 把编排函数注入队列模块，队列模块只持有一个可调用引用，从而避免直接 import。内嵌 Celery worker（workflow_worker_runtime）则可在单个 infrasrv 进程内以守护线程方式起停，无需额外容器。 The queue module (workflow_queue) needs to call the application layer's node orchestrator, while the application layer (workflow_app) imports the queue module — a circular dependency. The fix is runtime registration: at the end of its module load, the application layer calls register_inline_orchestrator(workflow_app._execute_node_orchestrator) to inject the orchestrator into the queue module, which holds only a callable reference and avoids the direct import. The embedded Celery worker (workflow_worker_runtime) can be started and stopped as a daemon thread inside a single infrasrv process, needing no extra container. 

# Celery 任务定义（worker 侧执行）/ Celery task (executed worker-side)
@celery_app.task(name=WorkflowTask.EXECUTE_NODE, bind=True, max_retries=3, acks_late=True)
def execute_node_task(self, run_id, node_id, loop_index=1):
    asyncio.run(workflow_app._execute_node_orchestrator(run_id, node_id, loop_index))
同一段编排逻辑，两条路。无论走 Celery worker 还是内联执行器，最终调用的都是同一个 _execute_node_orchestrator——它负责"加载运行与定义快照 → 执行节点 → 落库节点实例与日志 → 派发下一批节点"。降级只换执行载体，不换业务逻辑。One orchestration, two paths. Whether via a Celery worker or the inline executor, both ultimately invoke the same _execute_node_orchestrator — which "loads the run and definition snapshot → runs the node → persists the node instance and logs → dispatches the next batch". The fallback swaps the execution carrier, never the business logic.

模板表达式：节点之间引用数据Template expressions: referencing data between nodes

两套表达式语言贯穿引擎：Jinja2 做渲染与判断，JMESPath 做结构化提取。Two expression languages run through the engine: Jinja2 for rendering and decisions, JMESPath for structured extraction.

节点处理器执行前会构建一份模板上下文，把执行上下文摊平：可用 context.xxx 访问顶层字段、context.global_context.xxx 访问业务上下文、context.__runtime__.xxx 访问引擎运行时命名空间。ACTION 渲染请求、LOGIC / LOOP 求值条件用 Jinja2；TRANSFORM 可选 Jinja2 或 JMESPath。 Before a handler runs, it builds a template context by flattening the execution context: you can use context.xxx for top-level fields, context.global_context.xxx for business context, and context.__runtime__.xxx for the engine runtime namespace. ACTION request rendering and LOGIC / LOOP condition evaluation use Jinja2; TRANSFORM can choose Jinja2 or JMESPath.

__runtime__ 命名空间里最常用的三个键是 last_node_id / last_output / last_status——指向上一个执行完的节点，让"拿上一步的结果"变成一行表达式。完整的 node_outputs 字典则保留了每个节点的输出，可按节点 ID 回查。 The three most-used keys in __runtime__ are last_node_id / last_output / last_status — pointing at the most recently finished node, turning "use the previous step's result" into a one-line expression. The full node_outputs dict retains every node's output, queryable by node ID.

点语法的小处理。Jinja2 对 context.__runtime__ 这种双下划线键的点访问有属性解析限制，引擎会在渲染前把官方写法 context.__runtime__ 确定性地转换为 context['__runtime__']，所以你在配置里仍可以放心地用点语法书写。 A small dot-syntax fix. Jinja2 restricts attribute access on dunder keys like context.__runtime__, so before rendering the engine deterministically rewrites the official context.__runtime__ into context['__runtime__'] — meaning you can still safely write dot syntax in your config.

分布式锁：保护定义的并发改动Distributed lock: guarding concurrent definition edits

更新与回滚同一个工作流定义可能并发发生，引擎用 Redis 锁串行化它们。Updates and rollbacks of the same definition can happen concurrently; the engine serializes them with a Redis lock.

领域层在执行更新定义和回滚定义前，都会先拿一把按工作流维度的 Redis 锁，确保同一时刻只有一个改动在写。锁是阻塞获取的（带超时），拿不到锁会返回 409 让调用方稍后重试，而不是无限等待或并发覆盖。 Before updating or rolling back a definition, the domain layer first takes a per-workflow Redis lock, ensuring only one change writes at a time. The lock is acquired blockingly (with a timeout); failing to get it returns 409 telling the caller to retry later, rather than waiting forever or overwriting concurrently.

锁只保护"写定义"，不保护"运行"。运行实例靠各自的 definition_snapshot 隔离，天然不受定义改动影响（见概念页·版本与快照），因此运行路径不需要这把锁。锁的过期时间（5 分钟）也是一道兜底：万一持锁进程异常退出，锁会自动释放，不至于永久卡死后续改动。 The lock guards "writing the definition", not "running". Runs are isolated by their own definition_snapshot and are naturally unaffected by definition edits (see Concepts · Versioning), so the run path needs no lock. The 5-minute expiry is also a safety net: if a lock-holding process dies abnormally, the lock auto-releases instead of blocking future edits forever.
