Microstack 门面The Microstack facade

一个对象，三件事——其余细节都被它藏在身后。One object, three jobs — everything else hides behind it.

CancanMicrostack（位于 core/microstack.py）是一个很薄的门面（Facade）。它在内部把三个子系统组合在一起——AssetManager、ComposeBuilder、ServiceRunner——对外只暴露三个高层方法。 CancanMicrostack (in core/microstack.py) is a thin facade. Internally it composes three subsystems — AssetManager, ComposeBuilder, ServiceRunner — and exposes just three high-level methods.

方法Method作用What it does委托给Delegates to
build_compose()合成最终 compose 栈文件。Synthesize the final compose stack file.ComposeBuilder
export_asset()把包内资产复制到工作区。Copy a packaged asset into the workspace.AssetManager
run_service()启动一个内置服务入口。Start a bundled service entrypoint.ServiceRunner

价值在于：CLI 和你的代码都只需要认识 CancanMicrostack 这一个对象，不必关心三个子系统如何拼装。 The payoff: the CLI and your own code only ever need to know one object, CancanMicrostack — never how the three subsystems are wired together.

资产即包资源Assets as package resources

compose、DDL、脚本、运维前端随库一起发布——业务仓不再复制模板。compose, DDL, scripts and the ops UI ship with the library — no more copying templates into product repos.

所谓「资产」，是 cancan_microstack/assets/ 里随 Python 包一起发布的静态文件：docker compose 模板、DDL SQL、脚本、运维前端构建产物。AssetManager 用标准库 importlib.resources 访问它们，即使库被打成 wheel/zip 也能正确物化成真实文件路径。 An "asset" is a static file shipped inside the Python package under cancan_microstack/assets/: docker compose templates, DDL SQL, scripts, and the ops UI build output. AssetManager reaches them through the standard-library importlib.resources, so they materialize into real filesystem paths even when the library is packaged as a wheel/zip.

每个资产用一个稳定的逻辑名引用，例如 docker/docker-compose.infra.yml。调用方只认逻辑名，无需知道库内部文件的真实位置。 Each asset is referenced by a stable logical name, e.g. docker/docker-compose.infra.yml. Callers only deal with the logical name — never the file's real location inside the package.

为什么要 export？Docker 只看得见被挂载进容器的工作区路径，不会去 venv 的 site-packages 里找文件。所以模式是「先 export 到工作区，再让 Docker 挂载工作区路径」，既可分发、又符合容器只读卷的权限模型。 Why export at all? Docker only sees the workspace paths mounted into a container, not files buried in your venv's site-packages. So the pattern is: export into the workspace first, then mount the workspace path into Docker — distributable, and consistent with read-only container volumes.

工作区（workspace）The workspace

「你到底在哪个业务仓里跑」——一切相对路径与日志目录的基准。"Which product repo am I running in?" — the anchor for every relative path and log directory.

workspace root 是当前业务仓的根目录。它决定三件事：日志目录（server_log_data/）建在哪里、用户传入的相对路径以谁为基准、overrides 从哪里开始向上搜索。 The workspace root is your product repo's root directory. It decides three things: where the log directory (server_log_data/) is created, what relative paths resolve against, and where the upward search for overrides begins.

定位规则（runtime/workspace.py）按优先级：Detection rules (runtime/workspace.py), in priority order:

  1. 优先环境变量 CANCAN_WORKSPACE_ROOT，显式指定就用它。The CANCAN_WORKSPACE_ROOT environment variable wins if set.
  2. 否则从当前目录开始向上遍历，遇到任意标记文件/目录即认定为工作区根。Otherwise walk upward from the current directory; the first directory containing any marker is taken as the root.

标记文件任意命中一个即可： Any one of these markers is enough: server_log_datacmddocker-compose.infra.ymldocker-compose.services.ymlpyproject.tomlrequirements.txt

这正是它能在一个典型业务仓里「自动找到根目录」的原因——大多数项目至少有 pyproject.toml 或 requirements.txt。 This is exactly why it can "find the root automatically" in a typical repo — most projects have at least a pyproject.toml or requirements.txt.

免 Fork 覆盖（overrides）Fork-free overrides

只替换你想改的几个文件，其余仍复用库实现——不必 fork 主库。Replace only the few files you want to change; the rest still comes from the library — no fork required.

在工作区放一个 cancan_overrides/<service>/ 目录，只放你要替换的文件。运行时（runtime/overrides.py）用 pkgutil.extend_path 把该目录插到对应服务包搜索路径的最前面：import 时命中你的文件就用你的，否则回退到库自带实现。 Drop a cancan_overrides/<service>/ directory in your workspace containing only the files you want to replace. At runtime (runtime/overrides.py) pkgutil.extend_path prepends that directory to the service package's search path: on import, your file is used if present, otherwise it falls back to the library's implementation.

覆盖目录的发现也走两级：环境变量 CANCAN_OVERRIDE_ROOT 优先，否则从工作区向上寻找名为 cancan_overrides/ 的目录。 Discovery is two-tier as well: the CANCAN_OVERRIDE_ROOT environment variable takes priority, otherwise the loader searches upward from the workspace for a cancan_overrides/ directory.

目录树示例Directory layout

my-product-repo/
├── pyproject.toml              # workspace 标记 / workspace marker
├── docker-compose.services.yml
├── cancan_overrides/           # 覆盖根 / override root
│   └── infrasrv/               # 只覆盖 infrasrv / overrides for infrasrv only
│       └── config/
│           └── settings.py     # 命中你的；其余回退库 / yours wins; rest falls back
└── server_log_data/

这是典型的「插件 / 覆盖」机制：覆盖目录只需放被改的文件，未覆盖部分完全复用库本身。 This is the classic plugin/override pattern: the override directory holds only the changed files; everything untouched still comes straight from the library.

工作区优先、资产兜底Workspace-first, asset fallback

同名文件放工作区就用本地，否则用包内资产——一条规则贯穿配置解析。A same-named file in the workspace wins; otherwise the packaged asset is used — one rule across config resolution.

resolve_workspace_or_asset(relative_path, asset_logical)（runtime/resources.py）的规则非常直接：先看 <workspace_root>/<relative_path> 是否存在，存在就用工作区文件，不存在才回退到包内资产 AssetManager.resolve_path(asset_logical)。 resolve_workspace_or_asset(relative_path, asset_logical) (runtime/resources.py) is wonderfully direct: check whether <workspace_root>/<relative_path> exists; if it does, use the workspace file; only otherwise fall back to the packaged asset via AssetManager.resolve_path(asset_logical).

这让业务仓「放一个同名配置文件就能覆盖默认资产」，无需改动任何库代码。它和 overrides 是互补的两层：overrides 覆盖代码模块，本机制覆盖配置 / 资产文件。 This lets a repo "drop a same-named config file to override the default asset" without touching library code. It complements overrides as a second layer: overrides replace code modules, while this mechanism replaces config / asset files.
