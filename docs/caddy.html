Caddy 网关管理Caddy Gateway

通过 opsbffsrv 管理 Caddy 的路由、限流、证书、统计与访问日志分析。配置写在数据库里，由 sync 编译成 Caddy JSON 经 Admin API 下发，访问日志与流量则采集回库供查询。 Manage Caddy's routes, rate limits, certificates, statistics and access-log analysis through opsbffsrv. Config lives in the database, is compiled into Caddy JSON by sync and pushed via the Admin API, while access logs and traffic are collected back for querying.

路由管理Route management

把"哪个域名 + 路径 → 哪个上游"建模成路由记录，再批量同步给 Caddy。Model "which domain + path → which upstream" as route records, then batch-sync them to Caddy.

POSTcaddy/routes/{route_id}/toggle启用 / 禁用切换Enable / disable toggle
POSTcaddy/routes/sync把全部已启用路由编译并批量下发到 CaddyCompile all enabled routes and batch-apply to Caddy

两阶段生效。CRUD 只写配置库；真正改变 Caddy 行为的是 routes/sync。同步是分组替换：只重建本系统托管的路由组（cancan_managed_routes），保留 Caddy 上的非托管路由与兜底 404 路由，再整体 load 回 Caddy Admin API。若 Caddy 缺 WAF 模块，会自动剥除 WAF 处理器后重试。 Two-phase apply. CRUD only writes the config DB; what actually changes Caddy is routes/sync. The sync is a group replace: it rebuilds only this system's managed route group (cancan_managed_routes), preserves non-managed and the fallback 404 routes already on Caddy, and loads the whole config back via the Caddy Admin API. If Caddy lacks the WAF module, it strips the WAF handlers and retries automatically.

编译出的 Caddy JSON（单条路由）Compiled Caddy JSON (one route)

每条路由被 build_route_config() 编译成如下结构——match 匹配域名 / 路径，handle 依次串起 WAF、重写、反向代理（带主动健康检查）：Each route is compiled by build_route_config() into the shape below — match covers domain / path, and handle chains WAF, rewrite and reverse-proxy (with active health checks) in order:

{
  "group": "cancan_managed_routes",
  "match": [
    { "host": ["api.example.com"] },
    { "path": ["/api/v1/*"] }
  ],
  "handle": [
    { "handler": "coraza_waf",
      "directives": ["Include /etc/caddy/waf/coraza.conf"] },
    { "handler": "rewrite",
      "strip_path_prefix": "/api" },
    { "handler": "reverse_proxy",
      "upstreams": [{ "dial": "upstream.host:8080" }],
      "health_checks": {
        "active": { "path": "/internal/health",
                    "interval": "30s", "timeout": "5s" }
      } }
  ],
  "terminal": true
}

terminal 仅在 enable_https 且 force_https 时置 true；rewrite 处理器仅在配了路径前缀时出现；coraza_waf 仅在 enable_waf 时出现。terminal is set only when enable_https and force_https; the rewrite handler appears only with a path prefix; coraza_waf only when enable_waf.

核心字段Core fields

字段Field含义Meaning
match_type / match_pattern匹配维度：path / domain / ip / header / all，及对应模式Match dimension: path / domain / ip / header / all, plus its pattern
limit_type限流类型：request（请求数）或 bandwidth（带宽），默认 requestLimit type: request or bandwidth, default request
limit_value / limit_window窗口内允许的额度（>0）/ 时间窗（秒，≥1，默认 60）Allowance per window (>0) / window length (sec, ≥1, default 60)
limit_key限流主键：ip / header / cookie / path，默认 ipLimit key: ip / header / cookie / path, default ip
burst_size突发额度，默认 0Burst allowance, default 0
block_status_code / block_message触发限流时回应的状态码（默认 429）与消息体Status code (default 429) and body returned when throttled
whitelist_ips / blacklist_ips白 / 黑名单 IP 数组；提交时做 IP 合法性校验White / black IP arrays; validated for IP format on submit
priority规则优先级 0–1000，默认 100Rule priority 0–1000, default 100
限流由窗口 + 额度描述。本系统用 limit_value + limit_window（编译为 Caddy 的 max_events / window）表达"每窗口最多多少次"，并由 limit_key 决定按谁计数。白名单先于规则放行，黑名单始终拦截。 Limits are described by window + allowance. The system expresses "at most N per window" via limit_value + limit_window (compiled into Caddy's max_events / window), with limit_key deciding what to count by. The whitelist passes ahead of the rule; the blacklist always blocks.

字段Field含义Meaning
domain / alt_domains主域名（唯一）与备用域名（SAN）Primary domain (unique) and alt names (SAN)

issuer签发方名称Issuer name
issued_at / expires_at签发与过期时间，用于到期排序与提醒Issue / expiry timestamps, used for expiry sorting and alerts

auto_renew / renew_before_days是否自动续订（默认开）/ 提前续订天数（7–90，默认 30）Auto-renew (default on) / renew-ahead days (7–90, default 30)
acme_provider / acme_challenge_typeACME 提供方（letsencrypt 等）与挑战方式（http-01 等）ACME provider (letsencrypt etc.) and challenge type (http-01 etc.)
last_renew_attempt / last_renew_success / renew_error最近续订尝试 / 成功时间与失败原因Last renew attempt / success time and failure reason
私钥以明文 Text 存储。本表的 private_key_pem 不在应用层做加密（区别于运维登录的 TOTP secret——后者用 Fernet 加密）。生产中应配合数据库层加密 / 访问控制，避免私钥裸存。 Private keys are stored as plaintext Text. private_key_pem is not encrypted at the application layer (unlike the ops-login TOTP secret, which is Fernet-encrypted). In production, pair this with database-level encryption / access control so private keys are not stored in the clear.

续订流程Renewal flow

  1. 挑选到期候选Pick renewal candidates

    系统按 auto_renew=true 且 status=active 且 expires_at 落在 renew_before_days 内筛出候选证书。Candidates are filtered by auto_renew=true and status=active and expires_at within renew_before_days.

  2. 触发续订Trigger renewal

    手动 renew 或自动调度发起 ACME 续订。A manual renew or the auto-scheduler kicks off ACME renewal.

  3. 回写结果Record the result

    无论成败都回写 last_renew_attempt / last_renew_success / renew_error，供下次决策与排错。Success or failure, it writes back last_renew_attempt / last_renew_success / renew_error for the next decision and troubleshooting.

趋势接口按 stat_period（minute / hour / day / month）聚合，默认回看 hours=24；实时接口用 1 / 5 / 15 / 60 分钟多窗口回退，减少仪表板抖动。Top 排名默认窗口 1 小时、limit=20。每条统计记录覆盖请求数、成功 / 4xx / 5xx 分布、收发字节、响应时延（avg / p50 / p95 / p99）、WAF 拦截、限流命中、TLS / 非 TLS、独立 IP 数等指标。 Trend endpoints aggregate by stat_period (minute / hour / day / month), looking back hours=24 by default; realtime endpoints fall back across 1 / 5 / 15 / 60-minute windows to reduce dashboard flicker. Top rankings default to a 1-hour window and limit=20. Each stats record covers request counts, success / 4xx / 5xx split, bytes sent/received, latency (avg / p50 / p95 / p99), WAF blocks, rate-limit hits, TLS / non-TLS, unique IPs and more.

Suspicious-IP detection and error-pattern analysis aggregate over exactly these fields within a window.
