controllersrv — 容器控制服务Container Control Service

运行在宿主机上的守护进程，封装 Docker/Podman 双引擎容器编排（基于 dragonfly-container）。操作型请求走异步任务队列、立即返回流水号；查询型请求直读容器引擎。 A host-side daemon wrapping Docker/Podman dual-engine container orchestration (built on dragonfly-container). Operations enqueue onto an async task queue and return a serial number immediately; queries read the container engine directly.

概述Overview

为什么它是宿主机进程，而不是栈里的一个容器。Why it runs on the host, not as a container inside the stack.

controllersrv 的职责是启停整个微服务栈。它必须能直接对 Docker/Podman 守护进程下命令，因此它不在 compose 内——若把它放进栈里，它就无法在栈尚未运行时把栈拉起来，也无法在栈崩溃后重建容器。它以宿主机进程的形态长驻，监听 HTTP，是整个 Cancan 控制面的“底座的底座”。 controllersrv exists to bring the whole microservice stack up and down. It must issue commands directly to the Docker/Podman daemon, so it deliberately lives outside the compose stack — embedding it would mean it could not start the stack before the stack runs, nor rebuild containers after a crash. It runs as a long-lived host process listening over HTTP, the foundation beneath the rest of the Cancan control plane.

🔌

双引擎自动检测Dual-engine auto-detect

dragonfly-container 的 UnifiedExecutor 在 ExecutorFactory.create_unified_executor() 时自动探测 Docker 或 Podman，上层 API 无需区分引擎类型。dragonfly-container's UnifiedExecutor, created via ExecutorFactory.create_unified_executor(), auto-detects Docker or Podman so callers never branch on engine type.

⏳

异步任务模型Async task model

start / stop / restart 不阻塞 HTTP 连接：请求经校验后入队，立即返回 serial_number，由后台 Worker 串行执行，客户端再轮询任务状态。start / stop / restart never block the HTTP connection: validated requests enqueue and return a serial_number at once, a background worker executes them, and clients poll for status.

🧾

流水号防重Idempotency by serial

serial_number 是去重键：同一流水号重复提交会被识别为重复任务，避免误触发多次启停。The serial_number is a dedup key: re-submitting the same serial is detected as a duplicate, preventing accidental double start/stop.

🛡️

服务名白名单Service-name allowlist

操作前由 service_validator 校验服务名是否合法且可操作，非法或不可操作的服务名直接拒绝。A service_validator checks that each service name is valid and operable before any action; invalid or non-operable names are rejected outright.

限流：所有接口带本地限流（@limiter_local）。操作型与多数查询型为 10/minute，服务状态查询为 30/minute。所有响应统一为 APIResponse 信封（{ code, msg, data }），下表“关键字段”指 data 内字段。 Rate limits: every endpoint is locally rate-limited (@limiter_local) — 10/minute for operations and most queries, 30/minute for service status. All responses use the unified APIResponse envelope ({ code, msg, data }); the "key fields" columns below describe the data payload.

服务操作 APIService Operation API

启停重启整栈或指定服务，异步入队后返回任务凭证。Start / stop / restart the stack or named services; enqueue asynchronously and return a task handle.

方法Method 路径Path 用途Purpose 关键字段Key fields
POST/v1/controllersrv/service/start 启动指定服务，异步执行Start the named services, async 请求Req: serial_number, service_names[], params{} · 响应Resp: status(pending), serial_number, operation, service_names[]
POST/v1/controllersrv/service/stop 停止指定服务，异步执行Stop the named services, async 同 start，operation = stopSame as start; operation = stop
POST/v1/controllersrv/service/restart 重启（stop → start），异步执行Restart (stop → start), async 同 start，operation = restartSame as start; operation = restart

三者请求体共用 ServiceOperationRequest：serial_number（必填，去重）、service_names（必填，非空字符串列表）、params（可选附加参数）。成功响应是 EnqueueSuccessResponse，status 恒为 pending——它只代表“已入队”，真实结果须凭 serial_number 去任务管理 API 查询。 All three share the ServiceOperationRequest body: serial_number (required dedup key), service_names (required, non-empty string list) and params (optional extras). The success body is EnqueueSuccessResponse with status always pending — it confirms enqueue only; the real outcome is polled by serial_number via the Task Management API. 

# POST /v1/controllersrv/service/start
{
  "serial_number": "deploy-2026-0614-01",
  "service_names": ["infrasrv", "opsbffsrv"],
  "params": {}
}

任务管理 APITask Management API

查询入队任务的执行进度、历史与队列健康度。Inspect task progress, history and queue health.

方法Method 路径Path 用途Purpose 关键字段Key fields
GET/v1/controllersrv/task/status 按流水号查单个任务状态与结果Single task status & result by serial 请求Req: serial_number (query) · 响应Resp: task{}（含状态、结果、错误等）
GET/v1/controllersrv/task/list 列出任务，可按状态过滤List tasks, filterable by status 请求Req: status, limit(1–1000, def 100) · 响应Resp: tasks[], count
GET/v1/controllersrv/queue/stats 队列统计快照Queue statistics snapshot 响应Resp: stats{ queue_size, total_tasks, active_serial_numbers, status_counts, max_queue_size }
任务状态机：pending → running → success / failed。queue/stats 的 status_counts 按这些状态聚合计数，max_queue_size 来自 ServiceSettings.TASK_QUEUE_MAX_SIZE，队列满时入队会被拒绝。 Task lifecycle: pending → running → success / failed. The status_counts in queue/stats aggregates by these states; max_queue_size comes from ServiceSettings.TASK_QUEUE_MAX_SIZE, and enqueue is rejected when the queue is full.

容器 / 资源查询 APIContainer & Resource Query API

同步直读容器引擎：服务、容器、镜像、网络、卷与引擎健康。Synchronous reads off the container engine: services, containers, images, networks, volumes and engine health.

方法Method 路径Path 用途Purpose 关键字段Key fields
GET/v1/controllersrv/service/status 查指定服务运行状态Status of named services 请求Req: service_names[] (opt) · 响应Resp: success, services[]
GET/v1/controllersrv/service/list 列出所有受管服务List all managed services 响应Resp: success, services[]

GET/v1/controllersrv/containers/list 列出项目容器List project containers 请求Req: include_stopped (bool, def false) · 响应Resp: 容器列表
GET/v1/controllersrv/container/inspect 查容器详情Inspect one container 请求Req: container_id · 响应Resp: 容器详细信息
GET/v1/controllersrv/container/logs 取容器日志Fetch container logs 请求Req: container_id, tail(def 100), since (opt) · 响应Resp: 日志文本
GET/v1/controllersrv/images/list 列出镜像List images 响应Resp: 镜像列表
GET/v1/controllersrv/networks/list 列出网络List networks 响应Resp: 网络列表
GET/v1/controllersrv/volumes/list 列出卷List volumes 响应Resp: 卷列表

查询型接口（containers/list 起）直接走 UnifiedExecutor 的 API Executor 读取引擎状态，不经过任务队列，因此是同步即时返回的。 Query endpoints (from containers/list down) read engine state directly through the UnifiedExecutor API executor, bypassing the task queue, so they return synchronously.

  1. API 层（docker_control_api）：限流 → 校验 ServiceOperationRequest → 调 service_validator 验证服务名可操作 → 调用应用层。 API layer (docker_control_api): rate-limit → validate ServiceOperationRequest → run service_validator to confirm the names are operable → call the application layer.
  2. 应用层 DockerComposeApp.create_operation_task()：用 serial_number + 操作类型构造 Task 并 task_queue.enqueue()；重复流水号被识别为 duplicate，立即回 EnqueueSuccessResponse(status=pending)。 Application layer DockerComposeApp.create_operation_task(): build a Task from the serial + operation type and task_queue.enqueue() it; a duplicate serial is detected as such, then return EnqueueSuccessResponse(status=pending) at once.
  3. 领域层 / 队列：DockerComposeDomain 持有操作语义；TaskQueue 是进程内 asyncio.Queue，容量为 TASK_QUEUE_MAX_SIZE，满则拒绝入队。 Domain / queue: DockerComposeDomain holds operation semantics; TaskQueue is an in-process asyncio.Queue bounded by TASK_QUEUE_MAX_SIZE, rejecting enqueue when full.
  4. TaskWorker：后台循环 dequeue() 任务，按操作类型用 UnifiedExecutor.get_start_command / get_stop_command / get_restart_command 构造命令并执行，更新任务状态为 running → success/failed。 TaskWorker: a background loop that dequeue()s tasks, builds commands via UnifiedExecutor.get_start_command / get_stop_command / get_restart_command per operation type, executes them, and transitions task state running → success/failed.
  5. Executor：UnifiedExecutor 屏蔽 Docker 与 Podman 差异，真正对容器引擎下命令。客户端凭 serial_number 调 task/status 拿到最终结果。 Executor: UnifiedExecutor abstracts away the Docker/Podman difference and issues the actual engine commands. The client retrieves the final outcome via task/status using the serial_number.
